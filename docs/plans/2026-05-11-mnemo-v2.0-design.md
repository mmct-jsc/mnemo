# mnemo v2.0 -- Code Intelligence

**Status:** approved 2026-05-11
**Codename:** Code Intelligence
**Predecessor:** v1.1.1 (and v1.2, which ships first as a small, orthogonal release)

## 1. Goals, non-goals, scope

### Goal

Make mnemo **the only code-intelligence tool that also remembers why**.

Every code-intelligence tool today (GitHub Code Search, SourceGraph,
Cursor @-Codebase, GitNexus) answers *what* the code does and *how*
it's connected. None of them remember *why* it's there -- because
none of them have a memory layer. mnemo already has the memory
layer. v2.0 adds the code graph and -- critically -- the wiring
that links every function back to the decisions, incidents, retros,
and discussions that motivated it.

The headline capabilities are:

1. **Cross-stack sitemap.** "This React button calls this Express
   handler which queries this Postgres table" -- one graph
   traversal once Tier 3 framework extractors emit
   `Component -> Route -> Handler -> Service -> Table` edges.
2. **Decision provenance.** Hover any function in `/code` and see
   the linked `feedback_<incident>`, `plan_doc`, and commit messages
   that motivated it. Auto-generated "lessons learned" digest per
   project on the project dashboard. New `mnemo:why-is-this-here`
   skill walks the chain from code -> commit -> decision -> retro.
3. **Impact analysis with memory.** `mnemo:impact-analysis` returns
   the call-graph blast radius PLUS the `memory_feedback` notes
   anyone left about the affected functions ("don't refactor X
   without reading retro Y"). GitNexus's `impact` tool returns
   blast radius alone.

v2.0 also closes the structural gap that produced the v1.1.0 "Duyen"
bug: source kinds become explicit (`code_repo`, `docs_dir`,
`memory_dir`) and `mnemo source add` no longer silently misclassifies
a repo full of READMEs as memory.

### What ships in v2.0

1. **Explicit source typing.** Three kinds at launch: `code_repo`
   (tree-sitter), `docs_dir` (markdown harvest), `memory_dir`
   (existing semantics). Auto-router classifies on `mnemo source add`
   with a **dry-run preview** -- the user sees the proposed kind +
   file count + by-kind breakdown before anything writes to the DB.
2. **Tiered code graph.** Tree-sitter ingestion at three tiers:
   - Tier 1 (universal, 16 grammars): file / class / function / method
     / test / config nodes + resolved `imports` edges.
   - Tier 2 (semantic, 3 languages): cross-file `calls` edges for
     Python, TypeScript/JavaScript, Go via Stack-Graphs-style scope
     resolution.
   - Tier 3 (framework, 6 frameworks): sitemap edges for FastAPI,
     Express, React, Next.js, Django, Flask.
3. **Cross-stack composition.** Default is one-repo-one-project. New
   `linked_project` edge lets you compose multi-repo applications
   (`mnemo project link frontend-repo backend-repo --as "frontend-of"`)
   so retrieval graph proximity hops across the seam.
4. **`/code` UI family.** New page tree:
   `/code`, `/code/<project>`, `/code/<project>/file/<path>`,
   `/code/<project>/function/<id>`, `/code/<project>/sitemap`. Function
   detail uses lazy ego-network rendering (2 hops by default; click to
   expand 1 more hop) so a 5k-function repo never tries to render all
   at once.
5. **Decision provenance.** New `commit` node type ingested from
   `git log` per `code_repo`. New `motivated_by` / `closed_by` /
   `references_function` edges automatically link feedback /
   incidents / plan_docs to the specific commits and code functions
   they affected. UI: hover any function detail to see the linked
   "why we have this" sidebar. CLI / skill: `mnemo:why-is-this-here`
   walks the chain. Project dashboard gets an auto-clustered
   "lessons learned in this codebase" digest.
6. **Seven new skills** (was 5; +2 for provenance + impact).
   - `mnemo:explore-codebase` (flexible)
   - `mnemo:trace-call` (rigid)
   - `mnemo:trace-route` (rigid -- the sitemap walker)
   - `mnemo:explain-design` (flexible)
   - `mnemo:debug-with-code` (rigid -- mirrors today's `mnemo:debug`
     phases and folds in code-graph evidence at the investigate phase)
   - `mnemo:why-is-this-here` (rigid -- decision provenance walker)
   - `mnemo:impact-analysis` (rigid -- blast radius + memory the
     affected functions are tagged with)
7. **Watcher.** Per-file incremental re-parse with a 2.5s debounce
   window. OS filewatch (`watchfiles`) and the Claude Code
   PostToolUse hook feed the same dirty set. Edges originating from
   a changed file are rebuilt eagerly; edges terminating on it are
   re-resolved lazily on next query. Git commits also trigger
   incremental commit-ingestion (HEAD-aware staleness).
8. **Migration UX.** On first daemon start post-v2.0, the auto-router
   scans every existing `memory_dir` source. Anything that matches
   the heuristic for a different kind (e.g. has a `.git/` dir and
   non-frontmatter files) surfaces in a UI banner: "N source(s) look
   misclassified -- review them?" User accepts/rejects per source.
   Accept = drop the old nodes via cascade + re-ingest under the new
   kind. Reject = source stays untouched and is never re-prompted.
9. **50k file safety ceiling.** Auto-router refuses to scan a path
   whose recognized-source-file count exceeds 50k after applying
   default skip-dirs. User overrides with `--force` or narrows via
   `--include` / `--exclude`. Prevents Duyen-class accidents at v2.0
   scale.

### Non-goals (deliberate)

- **Chat surface.** Deferred to v3 ("Companion"). v2.0's skills run
  inside Claude Code; v3 builds the in-UI chat on top of v2's typed
  graph.
- **LSP integration.** Spawning `rust-analyzer`/`tsserver`/`pylsp`
  for cross-reference resolution breaks the local-first single-binary
  story. Tree-sitter + Stack-Graphs-style resolvers cover Tier 2 for
  the top 3 languages without it. LSP is a v2.x candidate behind an
  opt-in flag.
- **More than 16 grammars or 6 frameworks at launch.** Plugin grammar
  system is appealing but a v2.0.x add once we see what's missing in
  the wild.
- **Refactoring tools / write-back.** mnemo *reads* the codebase. It
  doesn't rename, restructure, or edit. That's the IDE's job.
- **Whole-repo render on `/code/<project>`.** Ego-network only. A
  unified everything-on-one-canvas view is a known scale failure for
  any non-trivial repo.
- **PR-diff annotation surface ("don't break me" warnings inline on
  PRs).** Slips to **v2.0.1** because it needs a separate trigger
  surface (GitHub Action / pre-commit hook / VS Code lens) outside
  the daemon's HTTP scope. Foundation is here in v2.0 -- the
  provenance edges are what the PR annotator queries -- but the
  trigger plumbing lands as a follow-on.
- **Memory-gap detection ("you keep asking about X but no node
  covers it").** Slips to **v2.x**, paired with v1.2's auto-tuner
  once enough labeled queries exist for the signal to be reliable.

### Success criteria

- `mnemo source add D:\Repository\Duyen` proposes `code_repo`,
  dry-run shows the file count, accepting indexes the repo
  successfully in under 5 minutes on a 5k-file repo with the default
  embed budget.
- "Where is `<function>` called from?" returns correct callers for
  Python, TypeScript/JavaScript, Go.
- "How does the `/api/users` route work?" cross-stack traversal
  surfaces the React component that calls it, the Express/FastAPI
  handler, the service layer, and any memory_feedback nodes that
  mention the route -- as a single ranked hit list.
- `/code/<project>/function/<id>` renders the focal function with
  its 2-hop ego-network in < 300 ms p95 on a medium repo.
- **"Why is `<function>` here?" returns** the commits that
  introduced + modified it, the `feedback_*` / `plan_doc` nodes that
  motivated those commits, and a one-paragraph synthesis citing each.
  This is the demo-able headline capability no other tool has.
- **Project dashboard auto-renders a "lessons learned" digest**
  clustering the project's `memory_feedback` nodes by theme, with
  per-cluster citations to the affected code functions. Render
  cost < 500 ms on a project with 30 feedback nodes.
- After upgrade, the auto-detect-on-upgrade UI banner offers to
  re-classify the user's Duyen registration (and any other
  pre-v2.0 misclassifications) without silent data loss.

### Retrieval path (extended)

```
                          Existing v1.1 path
   prompt -> intent classify -> vector top-K chunks
                              -> graph proximity from candidates
                              -> 6-term scoring (alpha..zeta)
                              |
                              v
              +---------------+---------------+
              |  v2.0 insertions:             |
              |  - code intent boost on      |
              |    type_priority (delta)     |
              |  - strict code isolation     |
              |    (no flooding memory)      |
              |  - linked_project hop        |
              |    for cross-stack edges     |
              +---------------+---------------+
                              |
                              v
                       compress to budget
                              |
                              v
                       emit hits + cite
```

### Invariants

- Code nodes default to `type_priority = low` so prose queries don't
  get flooded. Code intent classifiers (`where is X defined`, `who
  calls X`, `how does X work`, `trace this route`) boost it.
- Strict project isolation for code nodes by default (same hard-filter
  v1.1 introduced for BASE knowledge). `linked_project` edges let
  retrieval hop across cleanly-declared boundaries.
- BASE flag is *allowed* on code nodes for the rare cross-project
  utility-doc case. Not the default.
- Tree-sitter parsing is always-on for any registered `code_repo`.
  Stack-Graphs-style resolution (Tier 2) is async and lazy; if the
  resolver isn't done for a target language, the query falls back to
  Tier 1 structure.

### Auto-router

`mnemo source add <path>` (and `POST /v1/sources` without explicit
`kind`) goes through:

```python
def propose_kind(path: Path) -> tuple[str, str]:
    """Return (proposed_kind, confidence) for an unknown source path."""
    has_git = (path / ".git").is_dir()
    src_file_count = count_recognized_source_files(path, skip_default_dirs)
    md_frontmatter_count = count_md_with_frontmatter_type(path)
    md_no_frontmatter_count = count_md_without_frontmatter(path)

    if has_git and src_file_count > 0:
        return "code_repo", "high"
    if md_frontmatter_count >= 1:
        return "memory_dir", "high"
    if md_no_frontmatter_count >= 2 and src_file_count == 0:
        return "docs_dir", "medium"
    return None, "low"
```

Followed by **dry-run preview** which is the safety story:

```
$ mnemo source add D:\Repository\Duyen
Proposed kind: code_repo  (confidence: high)
Would index 1,247 file(s):
  - 412 python  (.py)
  - 188 typescript  (.ts, .tsx)
  - 76 yaml  (.yml, .yaml)
  - 31 markdown  (.md)
  - 540 other recognized source files
Would create approximately:
  - 1,247 code_module nodes
  - ~3,500 code_function nodes (estimated)
  - ~600 code_class nodes (estimated)

Proceed? [y/N/override kind]
```

Override path: `mnemo source add D:\Repository\Duyen --kind docs_dir`
forces the kind. Auto-router never writes the source row without
confirmation.

### Migration banner on first v2.0 daemon start

The daemon's lifespan hook runs the auto-router across every existing
`memory_dir` source and compares the proposed kind to the registered
kind. Any mismatch becomes a UI banner on the Sources page:

```
+--------------------------------------------------------------+
|  N source(s) look misclassified                              |
|  Pre-v2.0 only had one source kind; some of these might fit  |
|  the new code_repo / docs_dir kinds better.                  |
|                                                              |
|  [ Review ]                                                  |
+--------------------------------------------------------------+
```

Clicking Review opens a per-source dialog with the proposed
re-classification + the v1.1.1 cascade behavior wired in. Accept =
drop the old nodes + re-ingest. Reject = mark this source as
`migration_dismissed` in a new column so the banner stops asking.

For headless / scripting use of the same logic:

```bash

```

## 4. Code graph: tiered ingestion (Q3, Q7)

### Node types

```python
# Added to NODE_TYPES in store.py
CODE_NODE_TYPES = frozenset({
    "code_module",     # a file
    "code_class",
    "code_function",   # standalone function
    "code_method",     # class method
    "code_test",       # test function/class (heuristic: name starts test_,
                       # lives in tests/, or has @pytest decorator etc)
    "code_config",     # config file (json, yaml, toml) under a code_repo
    "code_route",      # framework route node (Tier 3)
    "code_component",  # React/Vue/Svelte component (Tier 3)
    "code_endpoint",   # framework endpoint URI (cross-stack anchor)
    "commit",          # NEW v2.0: git commit ingested from `git log`
                       #   per code_repo, used for decision provenance
})
```

### Edge relations

```python
# Added to EDGE_RELATIONS in store.py
CODE_EDGE_RELATIONS = frozenset({
    "imports",            # module A imports module B (Tier 1)
    "calls",              # function A calls function B (Tier 2)
    "defines",            # module defines class/function (Tier 1)
    "method_of",          # method belongs to class (Tier 1)
    "tests",              # test function tests target function (Tier 1+2)
    "routes_to",          # route -> handler (Tier 3)
    "renders",            # component -> child component (Tier 3)
    "queries_table",      # service/handler -> DB table (Tier 3 + heuristic)
    "linked_project",     # cross-project composition (Q6)
    # NEW v2.0 -- decision provenance:
    "references_function",# commit -> code_function it touched (auto from
                          #   git log + per-line diff attribution)
    "motivated_by",       # commit -> memory_feedback / plan_doc that
                          #   motivated the change (heuristic: commit body
                          #   mentions the doc's id/slug, or the doc was
                          #   updated in the same window)
    "closed_by",          # memory_feedback / plan_doc -> commit that
                          #   resolved it (inverse of motivated_by; also
                          #   detected from explicit "Closes #N" / "Fixes:"
                          #   commit trailers when issues are linked)
})

# All edges in v2.0 carry a `confidence` column (FLOAT, default 1.0).
# Tier 2 unresolved calls = 0.5; Tier 3 framework matches = 0.9;
# frontmatter-declared edges = 1.0; auto-inferred provenance edges
# (motivated_by, closed_by) = 0.6 unless commit body explicitly
# references the doc, which bumps to 0.9.
```

### Tier 1: universal structure (16 grammars at launch)

| Grammar | Notes |
|---|---|
| Python | Stdlib + pyi |
| TypeScript | Includes .tsx |
| JavaScript | Includes .jsx |
| Go | |
| Rust | Tier 1 only at launch; Tier 2 in v2.x |
| Java | Tier 1 only at launch |
| Ruby | |
| PHP | |
| C# | |
| Kotlin | |
| Swift | |
| Bash | Function-scoped only; flat scripts get one node |
| GDScript | For dogfooding against the SỢI repo |
| C | |
| C++ | |
| Markdown / JSON / YAML / TOML | Treated as docs/config under a code_repo |

Output per file:
1. One `code_module` node for the file itself, body = a 200-line head
   (or whole file if shorter) for the LLM to scan.
2. One node per top-level declaration: `code_class`, `code_function`.
3. One node per class method: `code_method` (with `method_of` edge to
   the class node).
4. `imports` edges resolved within the file's language scope (e.g.
   Python `from x.y import z` -> edge to `x/y.py::z`). Cross-file
   target resolution at Tier 1 is best-effort; falls back to a
   surface "unresolved" marker if the target isn't in the repo.

Distribution: a small core (Python, TS, JS, Markdown, JSON, YAML)
bundled in the wheel so first launch works offline. Other grammars
lazy-downloaded into `~/.claude/mnemo/grammars/<grammar>/` on first
`mnemo source add --kind code_repo` that needs them. SHA-pinned in
`daemon/mnemo/grammars/_pins.json`.

### Tier 2: semantic call graph (3 languages at launch)

Languages: **Python, TypeScript/JavaScript, Go**.

Implementation: per-language scope resolver inspired by GitHub's
[Stack Graphs](https://github.com/github/stack-graphs) approach but
written in Python to stay inside the existing daemon. The resolver
builds a scoped name graph from the tree-sitter AST, then resolves
each call-site identifier to its definition by walking scopes.

```
# Pseudocode
for each call_site in file:
    name = call_site.identifier
    scope = call_site.lexical_scope
    candidate = walk_scopes_until_definition(name, scope)
    if candidate:
        emit edge(call_site.parent_function, candidate.function, "calls")
    else:
        emit edge(call_site.parent_function, UNRESOLVED, "calls",
                  metadata={"name": name})  # falls back to lexical match
```

Rust / Java / C++ slot in at Tier 1 only for v2.0; their Tier 2
resolvers are a v2.x phase once we have data on how often "unresolved"
falls back lexically.

### Tier 3: framework extractors (6 at launch)

Each extractor is a small Python module in `daemon/mnemo/extractors/`
that pattern-matches over the tree-sitter AST for a language's typical
framework idioms.

| Extractor | Detects | Emits |
|---|---|---|
| `fastapi.py` | `@router.{get,post,...}` decorators, `APIRouter`, `app = FastAPI()` | `code_route` node per `@route`, `routes_to` edge route -> handler function |
| `express.py` | `app.{get,post,put,delete,...}(path, handler)`, `Router()` mounts | Same shape as FastAPI |
| `react.py` | JSX with `onClick={...}`, `useNavigate`, `<Link to=...>` | `code_component` + `routes_to`/`renders` edges between components |
| `nextjs.py` | `pages/api/<name>.ts` file, `pages/<name>.tsx` file, `app/api/<name>/route.ts` | `code_route` node from the path convention + edge to the file's default-export handler |
| `django.py` | `urls.py` `path()` / `re_path()`, `urlpatterns = [...]` | `code_route` node, edge to the named view |
| `flask.py` | `@app.route` / `@blueprint.route` decorators | Same as FastAPI |

A repo can match multiple extractors (e.g. Next.js front + FastAPI
back). Each extractor runs independently; emitted edges share the
`code_endpoint` URI node when paths match, which is what *creates*
the cross-stack `Component -> Route -> Handler` traversal.

### Storage strategy for code nodes

Code function bodies are commonly big and uninteresting to embed
verbatim. v2.0 stores:
- **`body`**: first 60 lines of the function (or whole if shorter) +
  a one-line `(... N more lines)` truncation marker. This is what
  the LLM sees in retrieval hits.
- **`source_path`**: `<file>:<start_line>-<end_line>` so the IDE can
  jump to the right place.
- **`description`**: one-liner extracted from docstring / leading
  comment.
- **`hash`**: SHA of the function's source for incremental detection.

Embeddings are computed on `description + body[:60 lines]` -- the
same surface the LLM sees. Avoids the "embed 5000-line function
verbatim" disaster.

## 5. Cross-stack composition (Q6)

Default: every `code_repo` source gets its own `project_key` derived
from path.

New edge type `linked_project` lets the user compose multiple repos
into one logical application:

```bash
# Register two repos
mnemo source add D:\Repos\my-app-frontend --kind code_repo
mnemo source add D:\Repos\my-app-backend  --kind code_repo

# Link them
mnemo project link D--Repos-my-app-frontend D--Repos-my-app-backend \
    --as "frontend-of"
```

Graph store representation: a `linked_project` edge between the two
projects' top-level `project` markers (created lazily when the first
code_repo source for a project_key is registered).

Retrieval honors `linked_project`: when running under
`isolation_mode=strict` with an active project, the hard-filter
admits nodes from any project reachable via a `linked_project` edge
(at the project level, not per-node). The cross-stack sitemap query
then naturally traverses the seam because both sides live in the
same retrieval scope.

UI: a "Linked projects" section on each `/code/<project>` page
showing the graph of inter-project relationships. CLI:
`mnemo project links` to list, `mnemo project unlink` to remove.

## 6. Decision provenance -- code that remembers why

This is v2.0's headline differentiator. Every code-intelligence tool
on the market shows *what* the code is and *how* it's connected;
none show *why*. mnemo already has the memory layer (typed
feedback / plan_doc / project notes). v2.0 adds the wiring that
auto-links those nodes to the specific functions and commits they
motivated.

### Ingest: `git log` as a first-class source

Every `code_repo` gains an automatic sub-ingest that walks `git log`
during the source's reindex pass. One `commit` node per commit, with:

```python
@dataclass
class CommitNode(Node):
    type = "commit"
    name = "<short_sha> <first line of subject>"
    description = "<author> <date> <one-line subject>"
    body = "<full commit message>"
    frontmatter_json = '{"sha": "<full>", "author_email": "...",
                        "ts": <unix>, "files_changed": N}'
    source_path = "<repo_path>@<sha>"
    source_kind = "code_repo"
```

Storage: bounded. We cap at the most recent 10k commits per repo by
default (configurable via `mnemo source patch <path>
--commit-limit N`). Older commits stay in git itself; if a user
needs deeper history they bump the cap and reindex.

### Auto-linking: `references_function` edges

For each commit, parse the diff hunks and map each touched line back
to its containing `code_function` / `code_method` / `code_module`
node (the file-line ranges we already track for code nodes make
this an O(N) join). Emit `references_function` edges from the
commit to each affected node, with `confidence` proportional to
the fraction of the function's lines the commit changed (a
one-character typo fix gets 0.3; a wholesale rewrite gets 1.0).

### Auto-linking: `motivated_by` / `closed_by` edges

Two heuristics, scored, both run on every commit:

1. **Explicit reference.** Commit body matches any registered
   memory_node's `name` (e.g. `feedback_mqtt_auth_flake`) or the
   slug from a `plan_doc`'s frontmatter. Confidence = 0.9.
2. **Co-temporal.** Within a 24-hour window of the commit, a
   memory_feedback / plan_doc node was added or updated AND its
   embedding has cosine >= 0.78 with the commit body. Confidence =
   0.6.

Best-rated edge wins for a given (commit, doc) pair. User can
promote / demote via the UI (sets `source="user"` on the edge so
heuristics don't override the manual call).

Trailers like `Fixes: #123`, `Closes: doc-name`, `Refs:
plan/2026-05-09-X.md` (configurable list in
`config.provenance.commit_trailers`) get parsed authoritatively for
`closed_by` edges (confidence = 1.0).

### Query: "why is this here?"

`mnemo:why-is-this-here` skill (rigid) walks:

```
code_function
   |
   v   references_function  (which commits touched me?)
commit (N)
   |
   v   motivated_by  (what decided this?)
memory_feedback / plan_doc / memory_project (M)
   |
   v   1-line synthesis with citations
```

Output format (CLI / chat-friendly):

```
auth.py::login (lines 42-58)

Most-recent material change: commit a1b2c3d (Alice, 2026-04-12)
  > "fix: short-circuit login on stale token (see retro 2026-04-10)"
  Motivated by: feedback_mqtt_auth_flake [mnemo:abc123]
    "Tokens flake under MQTT broker reprovision; short-circuit
    before broker check."

Earlier touches:
  - 11ac... (2026-03-30) implementation
  - 8842... (2026-02-14) renamed from `do_login`
    plan: 2026-02-10-auth-refactor.md [mnemo:def456]
```

The walk is bounded: 5 most recent commits per function by default;
configurable.

### Query: lessons learned digest

Project dashboard (`/code/<project>`) auto-renders a "Lessons learned
in this codebase" panel:

1. Pull all `memory_feedback` + `memory_project` nodes scoped to
   the project (`project_key == <project>` or BASE).
2. Cluster by embedding (k-means, k chosen by silhouette over [3, 8]).
3. Per cluster: pick the centroid node as the headline, pull the
   1-3 most-cited code_function nodes via `motivated_by` /
   `references_function` traversal.
4. Render:
   ```
   Lessons learned in petrolimex-edge:
   
   MQTT auth (4 incidents tracked)
     - Tokens flake under broker reprovision [mnemo:abc123]
     - Affected: auth.py::login, mqtt_client.py::connect
     - Lesson: short-circuit on stale token before broker check.
   
   Hard cooldown (3 incidents)
     ...
   ```

Re-clustered on every reindex with embedding changes. Cached for
the project_key.

### What this unlocks for v3 (chat)

When v3's chat surface lands, the killer query becomes:

> "Show me everything we've learned about MQTT auth, and where in
> the code those lessons live."

Today that's a hand-rolled grep across `~/.claude/projects/.../memory/`
+ a manual reading of git log. With v2.0's provenance edges in
place, it's a single `mnemo query` with a code intent boost. v2.0
ships the *data* the chat consumes; v3 ships the chat.

### v2.0.1 follow-on: PR-diff annotator

PR-time injection of provenance ("don't refactor X without reading
retro Y") is the natural follow-on but needs a trigger surface
outside the daemon's HTTP scope -- a GitHub Action, pre-commit
hook, or VS Code lens. v2.0 ships the data; v2.0.1 ships the
trigger. The annotator queries `mnemo:why-is-this-here` for each
function in the diff and surfaces high-confidence motivated_by /
closed_by edges as PR comments.

## 7. Code retrieval defaults (Q8)

Strict isolation + low default type_priority + intent-boosted +
opt-in BASE for shared code.

### Intent additions

New regex patterns in `daemon/mnemo/intent.py`:

```python
CODE_INTENT_PATTERNS = [
    (r"\bwhere (is|does)\s+\w+", "code-locate"),
    (r"\bwho calls\s+\w+", "code-callers"),
    (r"\bwhat does\s+\w+\s+(call|invoke)", "code-callees"),
    (r"\btrace\s+(this\s+)?(call|route|function)", "code-trace"),
    (r"\bhow does\s+\w+\s+work", "code-explain"),
    (r"\bsitemap|cross.?stack", "code-sitemap"),
    (r"\bimports?\b", "code-imports"),
]
```

When any of these fires, the scoring formula's `delta` (type priority)
term boosts `code_*` types over memory types for that query. The boost
is multiplicative on the type_priority weight, not on the term itself,
so prose memory still wins for prose-shaped queries even if a stray
code pattern matches.

### BASE for code

Allowed in frontmatter on a code-shaped *project_doc* (a markdown
note describing a shared coding pattern), but not on `code_function`
/ `code_class` -- the source body changes too often to be a stable
"universal lesson". Linted at parse time.

### Citation format

`[mnemo:<node_id>]` unchanged. UI adds an optional `file:line` anchor
shown next to the citation for code nodes, e.g.
`[mnemo:abc123]  src/auth/login.py:42-58`.

## 8. `/code` UI family (Q10)

Sub-routes under the existing FastAPI app:

| Route | Renders |
|---|---|
| `/code` | List of registered `code_repo` sources, per-repo node counts, "last indexed at" timestamps, link to project pages. |
| `/code/<project_key>` | Project overview: file-tree on the left, list of top-N central files (by edge degree) on the right, linked-projects sidebar. |
| `/code/<project_key>/file/<rel_path>` | File detail: imports list, defined classes/functions, who calls them, jump-to-IDE link (`vscode://file/<abs_path>`). |
| `/code/<project_key>/function/<node_id>` | Focal function detail: signature + first-60-lines body + ego-network Cytoscape canvas (2 hops by default). Each visible node has an "expand" button to load 1 more hop. |
| `/code/<project_key>/sitemap` | Cross-stack view: layered DAG of `Component -> Route -> Handler -> Service -> Table` for the project. Click any node to drill into its detail page. |

`/graph` (existing) stays exactly as it is -- memory + memory
cross-references only. Memory nodes that `mention` code nodes get a
sidebar link to the corresponding `/code/.../function/<id>` page.

### Rendering strategy

- File tree: HTMX lazy-loaded directory listings (don't render the
  entire tree at once for 5k-file repos).
- Function detail's ego-network: Cytoscape with `compoundLayout`,
  starts at the focal node, loads 2 hops via
  `GET /v1/code/ego/<node_id>?hops=2`. Each "Expand" click fetches
  `GET /v1/code/ego/<node_id>?hops=N&from=<focal>&except=<rendered>`.
- Sitemap view: vis.js layered DAG layout. Edges colored by relation
  (`routes_to` = blue, `renders` = green, `queries_table` = orange).

### Editor jump links

`source_path` for code nodes is `<abs_path>:<start>-<end>`. Both
`vscode://` and `cursor://` URL schemes are emitted; the UI picks
the right one based on a `~/.claude/mnemo/settings.json`
`preferred_editor` field. Default `vscode`.

## 9. New skills (Q11 + provenance/impact additions)

Seven skills under `skills/mnemo-*/SKILL.md` -- the five from the
Q11 brainstorming pass plus two added to capture the v2.0 standouts
(decision provenance + impact analysis, the latter informed by the
GitNexus comparison).

### `mnemo:explore-codebase` (flexible)

When: user wants to learn the shape of an indexed code_repo
("what's in this project", "show me the codebase", "how is this
organized").

Steps (loose):
1. Resolve `project_key` (from active project, or ask).
2. Pull top-N central nodes via `mnemo query "<project_key> overview"`
   with code intent boosted.
3. Render: structure summary (file count, language breakdown), top
   files by degree, entry points (mains, route definitions), open
   questions surfaced from `memory_feedback` cross-references.

### `mnemo:trace-call` (rigid)

When: "who calls X", "what does Y call", "trace this function".

Phases (strict):
1. Identify the target function -- match name + ask for disambiguation
   if multiple match.
2. Fetch callers via `GET /v1/code/edges?dst_id=<id>&relation=calls`.
3. Fetch callees via `GET /v1/code/edges?src_id=<id>&relation=calls`.
4. Render: focal node + caller list + callee list, each with
   `file:line` anchors and the first line of the calling expression.
5. Suggest follow-ups: "expand callers further?" / "trace the call
   chain to <function>?"

### `mnemo:trace-route` (rigid -- the sitemap walker)

When: "how does button X work", "where does API `/users` get
handled", "trace this UI action".

Phases (strict):
1. Identify the entry point: a `code_component` (frontend) or a
   `code_route` (backend). Disambiguate if multiple.
2. Walk the chain: `Component --renders--> Component
   --onClick--> Handler --calls--> Service --queries_table--> Table`
   (whichever edges apply for the project's stack).
3. Pull any `memory_feedback` nodes mentioning route segments along
   the path (e.g. "the `/api/users` endpoint had an auth bug in
   2026-03").
4. Render: linear chain with file:line anchors and any cited memory
   notes inline.
5. Done criterion: every step in the chain has a citation.

### `mnemo:explain-design` (flexible)

When: "how does auth work", "explain the X module".

Synthesize an answer from:
- The `code_module` body + any `project_doc` describing it.
- Top callers and callees of the module's exported functions.
- `memory_feedback` and `memory_project` nodes that mention the
  module's symbols.

Always emits `[mnemo:<id>]` citations and a "files referenced" list.

### `mnemo:debug-with-code` (rigid -- mirrors `mnemo:debug` phases)

When: "X is broken, why" -- but the user has indexed code_repo for
the project.

Phases (same shape as today's `mnemo:debug`):
1. Frame the symptom.
2. Pull priors via `mnemo query "<symptom>"` -- code intent boosted
   so code nodes mentioning the symptom surface first.
3. Trace the failing code path via `mnemo:trace-call` and/or
   `mnemo:trace-route` programmatically (read-only invocation).
4. Hypothesis list with citations to specific functions + commits.
5. Investigation: confirm the hypothesis with logs + reads.
6. Fix.
7. **Required:** RCA captured via `mnemo:add-knowledge` linking the
   `feedback_<incident>` node to the code_function nodes implicated.

### `mnemo:why-is-this-here` (rigid -- decision provenance walker)

When: "why is this code here", "what motivated this function",
"who decided this".

Phases (strict):
1. Identify the target node: `code_function` / `code_method` /
   `code_module`. Disambiguate if multiple match.
2. Walk `references_function` reversed -- pull the 5 most-recent
   commits that touched the node. Order by `confidence DESC`
   (larger structural changes first), then `created_at DESC`.
3. For each commit, walk `motivated_by` to the linked
   `memory_feedback` / `plan_doc` / `memory_project` nodes (any
   confidence >= 0.6 by default; configurable).
4. Render in the format documented in section 6: most-recent
   material change with motivation + earlier touches. Always
   emit `[mnemo:<id>]` citations.
5. Suggest follow-up: "see the full retro?" or "show all commits
   that touched this function".

Done criterion: every commit in the output is anchored to its
short SHA + author + date; every motivation has a citation.

### `mnemo:impact-analysis` (rigid -- blast radius + memory)

When: "what breaks if I change X", "what depends on Y", "is this
safe to refactor".

Phases (strict):
1. Identify the target function / class / module.
2. Walk reverse `calls` edges transitively up to a hop cap
   (default 4, configurable). Collect direct callers + tests +
   `routes_to` predecessors.
3. For each affected node, pull `memory_feedback` and
   `memory_project` nodes that `mention` it (or whose embedding
   has cosine >= 0.82 with the node's name + description).
4. Score risk:
   - **High** if any linked feedback has `severity >= 2` or any
     `closed_by` commit was reverted within 30 days of landing.
   - **Medium** if there are direct test coverage gaps (caller
     without a corresponding `tests` edge in the same file).
   - **Low** otherwise.
5. Render the blast radius as a depth-grouped list with risk
   level + memory citations per affected node.

Differs from GitNexus's `impact` tool: we layer institutional
memory on the call-graph blast radius. "X has 12 callers" is the
graph answer; "X has 12 callers AND we tagged 'don't refactor
without rebroadcast' three months ago" is the mnemo answer.

## 10. Watcher: per-file incremental + 2.5s debounce (Q12)

### Dirty set + debounce

```python
class DebouncedWatcher:
    DEBOUNCE_SECONDS = 2.5

    def __init__(self, store, scheduler):
        self._dirty: set[Path] = set()
        self._timer: Timer | None = None
        ...

    def on_change(self, path: Path) -> None:
        self._dirty.add(path)
        if self._timer:
            self._timer.cancel()
        self._timer = Timer(self.DEBOUNCE_SECONDS, self._flush)
        self._timer.start()

    def _flush(self) -> None:
        paths = list(self._dirty)
        self._dirty.clear()
        self._reparse_batch(paths)
```

`_reparse_batch` groups dirty files by source and runs the same
tree-sitter pipeline that does initial ingest, but for the touched
files only.

### Trigger channels

- **OS filewatch** via `watchfiles.awatch` on every enabled
  `code_repo` path. Fires for any editor's saves.
- **PostToolUse hook** in the Claude Code plugin -- already exists
  for memory dirs; matcher gains code-repo paths. Faster than OS
  filewatch for Claude Code edits.

Both channels call into the same `DebouncedWatcher.on_change`.

### Edge handling

- **Edges originating from a changed file** (e.g. an `imports` or
  `calls` edge whose `src` is in the file): rebuilt eagerly after
  re-parse.
- **Edges terminating on a changed file** (a `calls` edge whose
  `dst` is in the file): marked as needing re-resolution but not
  recomputed eagerly. On next query that traverses such an edge, the
  resolver runs lazily and caches the result.

This split makes a `Save All` of 50 files O(N) instead of O(N^2).

### Rename detection

Out of scope for v2.0. `watchfiles` emits delete+create on rename;
v2.0 treats this as orphan + new node (cascade fixes the orphan via
the v1.1.1 logic). Content-hash-based rename detection is a v2.0.x
candidate.

### Branch switch

A `git checkout` that flips 100s of files at once produces one
debounced batch. Batched re-parse with the embedder warmed up:
~5 ms/chunk + parse overhead. For a 50-file Python checkout that's
roughly 1-2 seconds elapsed.

## 11. Performance budget + safety (Q13)

### Targets

| Tier | Files | Init scan | Query p95 | RAM | save -> graph |
|---|---|---|---|---|---|
| Small | < 500 | 30 s | 200 ms | 200 MB | < 3 s |
| Medium | 500-5k | 5 min | 500 ms | 400 MB | < 5 s |
| Large | 5k-50k | "best effort" | "best effort" | "best effort" | "best effort" |

- **Hard SLA**: query latency p95 ≤ 500 ms on Small + Medium.
- **Soft SLA**: init time. One-time cost; users accept a coffee break.
- Init scan dominated by embed cost (5 ms/chunk x N chunks); we
  parallelize tree-sitter parsing across cores.

### Default skip rules

Applied by the auto-router and by per-source default include sets.

**Skip directories** (`exclude` glob `**/<name>/**`):
`node_modules`, `.venv`, `venv`, `__pycache__`, `target`, `build`,
`dist`, `out`, `.next`, `.git`, `.idea`, `.vscode`,
`coverage`, `htmlcov`, `.pytest_cache`, `.mypy_cache`,
`.ruff_cache`, `.terraform`, `vendor`.

**Skip files**:
- Any single source file > 5 MB. Probably generated / minified.
- Files matching `*.min.{js,css}`, `*.bundle.{js,css}`, `*.map`,
  `*.lock`, `*.snap` (jest snapshots).

User overrides via per-source `include` / `exclude` patterns.

### 50k file safety ceiling

```python
def precount_and_check(path: Path, max_files: int = 50_000) -> int:
    """Return the count of recognized source files; raise if over the cap."""
    n = count_files_matching_default_includes(path, default_skip_dirs)
    if n > max_files:
        raise TooManyFilesError(
            count=n, max=max_files,
            message=(
                f"{path} has {n} recognized source files after default "
                f"skip-dirs, which exceeds the {max_files} ceiling. "
                f"Add --force to override, or narrow with "
                f"--include/--exclude patterns."
            ),
        )
    return n
```

`mnemo source add` and the auto-router call this before any
ingestion. UI surfaces it as a banner with the file count + a
"continue anyway" button. CLI uses `--force`.

### Risk register

- **Tree-sitter grammars bloat the wheel.** Mitigation: bundle a
  small core (Python/TS/JS/MD/JSON/YAML); rest lazy-downloaded.
- **Stack-Graphs-style resolver is wrong for our language matrix.**
  Mitigation: Tier 2 is a *boost*, not a requirement. Unresolved
  calls fall back to lexical matching for the LLM to disambiguate.
  Edge `confidence` carries the uncertainty into retrieval.
- **Big repos hang the watcher.** Mitigation: 2.5 s debounce, 50k
  file ceiling, default skip-dirs.
- **Commit ingestion bloats the store on a repo with 100k commits.**
  Mitigation: default cap of 10k most-recent commits per repo;
  configurable. Older commits stay in git; deeper history requires
  explicit opt-in.
- **`motivated_by` auto-linker false positives** (commit looks
  co-temporal with an unrelated retro). Mitigation: confidence
  scoring (0.6 vs 0.9) + UI lets user demote / promote with
  `source="user"` override that heuristics respect.
- **Migration banner false positives.** Mitigation: dismissal is
  per-source and persists; never auto-applies.
- **`/code/<project>/function/<id>` ego-network blows the browser
  on a hub node** (e.g. `print` in Python, 5,000 callers).
  Mitigation: 2-hop default + per-hop hard cap of 200 nodes; "200+
  more, expand?" footer.
- **Cross-stack endpoint node collisions** (two repos use `/api/v1`).
  Mitigation: `code_endpoint` nodes are namespaced by the registering
  source's project_key; cross-stack edges only form when explicit
  `linked_project` is in place.

### Estimate

~3.5 weeks of focused work. Larger than v1.1 (~2 weeks for 11 phases)
because Tier 2 resolvers, framework extractors, and the provenance
auto-linker are each their own mini-project. v1.2's auto-tuner is
reusable in v2.0 without modification -- it's generic over the
6-term weights (and now over the edge-confidence multiplier).

## 15. Sequencing with v1.2

v1.2 ships first as a **small, orthogonal release** (~1.5 weeks, 8
phases). Its `feedback_event` table + coordinate-descent auto-tuner +
MMR re-ranker + housekeeping (drop legacy 308 redirects + `POST
/v1/nodes`) land independently.

v2.0 then starts from `main` post-v1.2 and inherits:
- The auto-tuner, which is generic over the 6-term weights. When v2.0
  introduces new term weights specific to code (e.g. a `code_proximity`
  multiplier or a separate type_priority slot for code intents), the
  tuner picks them up without code changes.
- MMR re-ranking, which prevents code result lists from devolving to
  five paraphrases of the same function.
- `POST /v1/nodes`, useful for HTTP-driven `linked_project` setup
  scripts.

No content from v1.2's roadmap is *required* by v2.0, so if v1.2
slips, v2.0 can still ship -- it just doesn't benefit from the tuner
on day 1.
