> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans. RIGID: superpowers:test-driven-development, superpowers:verification-before-completion. Live-verify the settings UI per `feedback_reproduce_user_exact_scenario` (real `/settings/chat`, computed-style/DOM probes — not screenshots). **HARD CONSTRAINT:** do NOT physically merge the v1.2 retrieval-tuning route/context (gotcha 9 / regression-guarded) — unify the *nav + shell only*, keep `/settings` and `/settings/chat` as separate routes/templates.

**Goal:** Make the provider/key settings UI **registry-driven** (reads C2's `GET /v1/providers`): every registered provider appears automatically with a real model **picker** (from `known_models`), a read-only **key-resolution-tier** indicator (env/dotenv/keychain/file), and a **delete-key** action — plus one unified settings information-architecture (a single nav entry → tabbed shell spanning retrieval ∪ providers ∪ companion) without merging the regression-guarded retrieval route.

**Architecture:** Settings panels become schema/registry-derived rather than `cfg.providers`-derived. The provider tab fetches `GET /v1/providers` (the C2 registry) for the provider list + `known_models`, and `GET /v1/settings` for per-provider `has_key`/`model`/**new** `key_tier`. New `keys.resolve_api_key_tier()` + `keys.delete_api_key()` + `DELETE /v1/settings/providers/{name}/key` follow the existing `set_api_key`/`POST /v1/settings/providers` invariant (keys via keychain/file, never JSON-persisted; model via `config.update`). Nav unification is server-rendered cross-links / a shared tab strip — **routes and contexts stay physically separate**.

**Tech Stack:** FastAPI + Pydantic (`SettingsOut` extension, new DELETE route), Jinja2 + Alpine.js (`chat_settings.html`), `keyring` (optional, already a dep). pytest + TestClient.

```python
def resolve_api_key_tier(provider: str, *, dotenv_path=None) -> str | None:
    """Where would resolve_api_key(provider) get its value from?
    Read-only mirror of the resolve_api_key ladder; returns the tier
    name, never the secret."""
    var = ENV_VAR.get(provider)
    if var and os.environ.get(var):
        return "env"
    if _dotenv_has(provider, dotenv_path):   # factor the .env branch of resolve_api_key
        return "dotenv"
    if _keyring_key(provider):
        return "keychain"
    if _plaintext_key(provider):
        return "file"
    return None


def delete_api_key(provider: str) -> None:
    """Remove a stored key (keychain + plaintext fallback). Env/.env
    tiers are not ours to delete -- callers surface that read-only."""
    _keyring_delete(provider)          # add: keyring.delete_password("mnemo", f"provider:{provider}") guarded by the optional import
    _plaintext_delete(provider)        # add: load keys.json, pop provider, rewrite 0600 (no-op if absent)
```

Factor the existing `.env` discovery used inside `resolve_api_key:112-116` into `_dotenv_has(provider, path)` so tier detection and resolution share one implementation (DRY — do not duplicate the ladder).
