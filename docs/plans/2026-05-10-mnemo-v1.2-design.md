# mnemo v1.2 — Learning to Listen

**Status:** approved 2026-05-10
**Codename:** Learning to Listen (personalization / learning release)
**Predecessor:** v1.1.0

## 1. Goals, non-goals, scope

### Goal

Make mnemo *adapt to the user* over time. Every query the user runs
and every hit they accept (or reject) becomes signal that nudges
the scoring weights toward the user's actual taste. Plus a small
re-ranking pass that diversifies the top-K so the user sees five
distinct angles instead of five paraphrases of the same node.

### What ships in v1.2

1. **Feedback collection.** Thumbs-up / thumbs-down on every
   retrieval result (UI + CLI). Implicit re-query detection (within
   N minutes of a similar prompt). New `feedback_event` table. New
   `POST /v1/feedback` endpoint.
2. **Auto-tuner.** New `mnemo retune` CLI + UI button. Coordinate
   descent over the six scoring weights, optimizing MRR on the
   labeled audit-log queries. Shows the diff and asks for
   confirmation before committing.
3. **MMR re-ranking.** Top-K results pass through a Maximal Marginal
   Relevance pass that penalizes near-duplicates of already-picked
   candidates. Uses existing MiniLM embeddings — no new model, no
   latency cost.
4. **Housekeeping.** Drop the legacy `/health` `/sources` etc. 308
   redirects (promised for v1.2 in v1.1's CHANGELOG). Add
   `POST /v1/nodes` so the VS Code "Add Note" command (and any
   other HTTP client) can create memory entries directly without
   writing files first.

### Non-goals

- No cross-encoder re-rank (deferred to v1.3 — would add ~1 s
  latency).
- No SaaS-side ingesters (Notion / Confluence / GitHub / Slack —
  deprecated 2026-05-10).
- No VS Code Marketplace publish (deprecated 2026-05-10).
- No JetBrains plugin (deprecated 2026-05-10).
- No nightly auto-retune (v1.3 candidate after the on-demand
  version proves itself).
- No NDCG@K objective (MRR is enough at small data scale).
- No embedding cache (marginal value at ~5 ms per embed).

### Success criteria

- A new user, after running ~100 queries with thumbs up/down on a
  few, sees `mnemo retune` propose weight nudges of magnitude ≤ 0.1
  that improve MRR by ≥ 5 % on validation. Diff is human-readable;
  rollback is one click.
- The top-5 of any query has zero near-duplicate pairs (cosine
  > 0.92) at MMR's λ = 0.7.

## 2. Architecture

### Retrieval path with v1.2 insertion

```
                Existing v1.1 path
   prompt -> intent -> vector top-40
                    -> graph proximity
                    -> 6-term scoring
                    -> top-K
                                |
                                v
                +-----------------------------+
                |     v1.2 INSERT POINT       |
                |  MMR diversify top-K -> K   |
                +-----------------------------+
                                |
                                v
                    compress to budget
                                |
                                v
                    emit hits + cite
```

### Feedback loop

```
            UI / CLI / chat
                  |
                  v thumbs-up / thumbs-down
        POST /v1/feedback
                  |
                  v
       feedback_event table
                  |
                  v   (passively, on every query)
       audit + similar-prompt detector
                  |
                  v   inferred -0.5 if user re-queries close-prompt
       feedback_event table
                  |
                  v   (on demand: `mnemo retune`)
       coordinate-descent optimizer
                  |
                  v
       proposed weight diff (alpha 0.40 -> 0.43, ...)
                  |
                  v   user-approved
       config.scoring.* updated -> next query uses new weights
```

### Invariants

- Feedback collection is **always-on but optional**. The user is
  never required to thumb anything; the system works as v1.1 if no
  feedback ever arrives. Auto-tuner just refuses to retune until
  enough labeled data exists (configurable threshold, default 30
  labeled queries).
- The auto-tuner is a **read-only** consumer of `feedback_event`
  and `queries`. It writes back only to `config.scoring.*` and
  only when the user confirms.
- MMR runs in-process, in-memory, ~1 ms overhead. No new
  dependencies.
- Inferred re-query signal runs **at write time**: if the new
  query has cosine ≥ 0.85 with any query in the last 5 minutes,
  write a `signal=-0.5, reason='inferred_requery'` row against
  the original query's hits.

## 3. Feedback collection

### Schema

```sql
CREATE TABLE feedback_event (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  query_id    TEXT NOT NULL,        -- joins to queries.id
  node_id     TEXT NOT NULL,        -- the hit being scored
  signal      REAL NOT NULL,        -- +1.0 thumbs-up; -1.0 thumbs-down;
                                    -- -0.5 inferred from re-query
  reason      TEXT NOT NULL,        -- 'thumbs_up' | 'thumbs_down' |
                                    -- 'inferred_requery' | 'cite_copied'
  created_at  INTEGER NOT NULL,
  FOREIGN KEY (query_id) REFERENCES queries(id) ON DELETE CASCADE,
  FOREIGN KEY (node_id) REFERENCES nodes(id) ON DELETE CASCADE
);

CREATE INDEX idx_feedback_query ON feedback_event(query_id);
CREATE INDEX idx_feedback_node  ON feedback_event(node_id);
CREATE INDEX idx_feedback_ts    ON feedback_event(created_at DESC);
```

The same row format covers all four signal sources so the
optimizer's input is uniform. `cite_copied` is wired by the UI's
"Copy citation" button (already exists in v1.1; v1.2 just records
it). The `reason` field reserves space for future signal types.

### HTTP surface

| Method | Path | Purpose |
|---|---|---|
| POST | `/v1/feedback` | `{query_id, node_id, signal, reason}`. Idempotent — same `(query_id, node_id, reason)` triple updates instead of duplicates. |
| GET | `/v1/feedback?query_id=...` | List events for a query (debug + retune review). |

### UI

Each hit on the search results page (and inside the graph side
panel and `_search_results.html`) gains two small icon buttons:
`↑` and `↓`. Click sends the POST. State flips to "filled" so the
user sees their choice. (Optional 1-line "tell us why?" expanding
inline on thumbs-down — out of v1.2 scope.)

### CLI

`mnemo query` gains `--with-feedback`: after rendering hits,
pause for `[1-N] thumbs up; -[1-N] thumbs down; q quit`. POSTs
the result. Matches the daemon's idempotency contract.

### Implicit re-query inference

On every `POST /v1/query`, after writing the audit row:

```
recent = queries.recent_within(window_seconds=300)
for q in recent:
    if cosine(query_emb, q.embedding) >= 0.85:
        for hit in q.retrieved_ids[:3]:
            insert feedback_event(query_id=q.id, node_id=hit,
                                  signal=-0.5, reason='inferred_requery')
```

Cheap (10-20 rows in the recent window), runs in the same
transaction as the audit write. **Note**: storing query
embeddings in the audit table is a small v1.2 schema addition
(`queries.embedding BLOB NULL`) — the existing audit log doesn't
keep them.

## 4. Auto-tuner internals

**Module:** new `daemon/mnemo/retune.py`. CLI entry: `mnemo
retune`. UI button on `/settings`.

### Inputs

- The `queries` audit log.
- The `feedback_event` table.
- Current `config.scoring` (the search start point).
- Threshold: only retune when there are ≥ 30 labeled queries
  (queries with at least one feedback row). Configurable via
  `config.retune_min_queries`.

### Train / validation split

Sort labeled queries by `created_at`; oldest 80 % is train,
newest 20 % is validation. Time-ordered split avoids the
optimizer overfitting to recent feedback patterns.

### Objective: MRR

For a candidate weight vector `w`:

```python
def mrr(w, queries):
    total = 0.0
    n = 0
    for q in queries:
        rescored = rescore_with_weights(q, w)
        for rank, hit in enumerate(rescored, start=1):
            sig = best_feedback_signal(q.id, hit.node_id)
            if sig > 0:
                total += 1.0 / rank
                break
        n += 1
    return total / max(n, 1)
```

`rescore_with_weights` is the same per-candidate scoring formula
already in `retrieve.py`, factored out so it can be called with
arbitrary weights.

### Optimizer: coordinate descent

```python
nudges = [-0.10, -0.05, +0.05, +0.10]
keys   = ['alpha', 'beta', 'gamma', 'delta', 'epsilon', 'zeta']
weights = current_weights.copy()
best_score = mrr(weights, validation)
for pass in range(MAX_PASSES):  # MAX_PASSES = 4
    improved = False
    for k in keys:
        for n in nudges:
            candidate = weights.with_change(k, weights[k] + n)
            if not (0 <= candidate[k] <= 1.0):
                continue
            s = mrr(candidate, validation)
            if s > best_score + EPS:  # EPS = 0.001
                weights = candidate
                best_score = s
                improved = True
    if not improved:
        break
```

Total cost: ≤ 6 × 4 × 4 = 96 evaluations × ~250 ms each = ~25 s
on a 500-row dataset. Capped at 60 s wall-clock.

### Output

```python
@dataclass
class RetuneReport:
    proposed: dict[str, float]
    current: dict[str, float]
    diff: dict[str, float]
    train_mrr_before: float
    train_mrr_after: float
    val_mrr_before: float
    val_mrr_after: float
    iterations: int
    train_size: int
    val_size: int
    log: list[str]
```

The CLI prints the diff + MRR delta + log. The UI shows the same;
"Apply" PUTs to `/v1/config`. "Discard" closes the panel.

## 5. MMR re-ranking

**Module:** new `daemon/mnemo/rerank.py`. Plugged into
`retrieve.query` between scoring and compression.

### Algorithm

At each step, pick the candidate that maximizes:

```
score_mmr(c) = lambda * relevance(c)
             - (1 - lambda) * max_sim(c, already_picked)
```

where `relevance(c)` is the existing 6-term score (normalized to
`[0, 1]`) and `max_sim(c, picked)` is the cosine similarity of
`c`'s best chunk embedding to any already-picked candidate's
chunk embedding.

### Wired in

```python
# retrieve.query, after scored.sort(key=lambda h: -h.score)
top_pool = scored[:max(k * 2, 20)]
top = mmr_select(top_pool, k=k, lambda_=cfg.mmr_lambda)
```

### Default lambda

`0.7` — leans toward relevance, with enough diversity penalty to
nuke near-duplicates. Settable via `config.mmr_lambda` (range
`[0.0, 1.0]`; 1.0 = no diversity = today's behavior; 0.0 = pure
diversity, ignores relevance).

### Cost

O(k × pool_size) cosine ops on already-cached chunk embeddings.
With pool=20, k=5: 100 ops × ~5 µs each = ~0.5 ms total. Negligible.

### Edge cases

- Pool < k: return the pool as-is, no diversification needed.
- Tied scores: deterministic tiebreak (first-pick wins) preserves
  the existing ordering.
- `lambda_ = 1.0`: bypass MMR entirely (saves the 0.5 ms).

### Why not cross-encoder

Cross-encoder re-rank would also fix near-duplicate noise but
adds ~1 s of latency on a default k=5 query. MMR fixes ~80 % of
the problem (visible duplicates) for ~0.1 % of the cost.
Cross-encoder is the right v1.3 add when paired with a
quality-first scoring mode.

## 6. Housekeeping

### Drop legacy 308 redirects

Promised in v1.1's CHANGELOG.

```python
# daemon/mnemo/server.py — remove _LegacyRedirectMiddleware entirely.
# Remove `app.add_middleware(_LegacyRedirectMiddleware)`.
# Update test_v1_redirect.py: every test that expected 308 from
# /health, /sources, etc. now expects 404 (route not found).
# Replace with a single test_legacy_paths_404 verifying the cleanup.
```

Migration: any adapter still on legacy paths breaks. The contract
from v1.1 was that the redirects would survive *only* the v1.1
series; v1.2 lands the cut. The `X-Mnemo-Api-Version` header has
been signalling "use /v1/" for the whole v1.1 cycle, so adapters
had time to migrate.

### `POST /v1/nodes`

Direct memory creation over HTTP. Useful for the VS Code "Add
Note" command (currently opens the dashboard as a workaround) and
for any future HTTP client.

```python
@v1.post("/nodes", response_model=NodeOut)
def create_node(body: NodeCreateIn, s: Store = Depends(get_store)) -> NodeOut:
    node = Node.new(
        type=body.type,
        name=body.name,
        body=body.body,
        source_path=body.source_path or f"http://api/{uuid.uuid4().hex}",
        source_kind=body.source_kind or "memory_dir",
        description=body.description,
        project_key=body.project_key,
        base=body.base or False,
    )
    s.upsert_node(node)
    if embedder is not None:
        _embed(s, node, embedder)
    return NodeOut.from_node(node)
```

`source_path` defaults to a synthetic `http://api/<uuid>` so the
existing reconcile + de-dup logic still works. The watcher won't
touch HTTP-created nodes since they're not on the filesystem.

`NodeCreateIn`:
- `type`: required, must be a valid `NODE_TYPE`
- `name`: required, non-empty
- `body`: required
- `description`, `source_path`, `source_kind`, `project_key`,
  `base`: optional

VS Code's "Add Note" wires through to this endpoint in the same
v1.2 commit.

| 8 | Release | docs + version files | `chore(release): v1.2.0` |

### Risk register

- **Auto-tuner overfits to early data.** Mitigation: 30-query
  threshold; train/val split shows you the val MRR; user must
  approve the diff. Worst case: rollback by re-PUTting the old
  weights.
- **Inferred re-query false positives.** Two close-but-distinct
  queries within 5 min get spurious -0.5 signals. Mitigation:
  explicit thumbs override (positive thumbs cancel inferred
  negatives via `best_feedback_signal`). Track FP rate in retune
  logs.
- **Dropping 308 redirects breaks unmigrated adapters.**
  Mitigation: v1.1's whole release cycle has been telling
  adapters via `X-Mnemo-Api-Version` to migrate. Document the
  breakage in CHANGELOG.
- **`POST /v1/nodes` lets clients spam the store.** Daemon is
  127.0.0.1-only — same trust boundary as the existing
  `PUT /v1/nodes/{id}`. No new attack surface.
- **MMR diversifies away the user's intended hit.** Mitigation:
  λ = 0.7 leans toward relevance; MMR runs on top of (not instead
  of) the existing scoring. User can set `mmr_lambda = 1.0` to
  disable.

~1.5 weeks of focused work. Less than v1.1.
