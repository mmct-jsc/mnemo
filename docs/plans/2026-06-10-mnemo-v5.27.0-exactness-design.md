# mnemo v5.27.0 -- exactness: BM25 + RRF-style lexical + exact-name boost (workstream C, part 2)

> DoD-first design doc (2026-06-10). Measured against the v5.26.0 baseline
> (`mnemo eval`: n=14 hit@1=0.14 hit@5=0.21 mrr=0.16). Per pipeline #21;
> rides with release/5.27.0.

## 1. Goal

Beat the baseline on the SELF set. The recorded misses share one shape:
"where/how is X" questions return giant memory/handover PROSE nodes while
the answering CODE node (whose name/body literally contain the terms)
never surfaces. Embeddings dilute on long bodies; the existing
`_lexical_score` (substring fraction, zeta=0.2) only scores nodes that are
ALREADY candidates (vector ∪ graph) -- a lexically-perfect code node that
isn't in the vector top-40 is invisible.

## 2. Components

### A. SQLite FTS5 lexical channel (store)

- New virtual table `nodes_fts` (id UNINDEXED, name, description, body) --
  body capped at the existing 32 KB lexical slice so a 500 KB plan_doc
  doesn't bloat the index. Created in `_ensure_schema`
  (`CREATE VIRTUAL TABLE IF NOT EXISTS ... USING fts5`); backfilled once
  when empty while `nodes` is not.
- Synced in `upsert_node` (delete+insert) and `delete_node`; plus
  `rebuild_fts()` as the consistency backstop callable after bulk
  operations (reindex tail, source cascade).
- `bm25_search(query_text, k) -> list[(node_id, rank_position)]`: tokens
  via the retrieve tokenizer, FTS-safe MATCH (`"tok1" OR "tok2"`), ordered
  by `bm25(nodes_fts)`. Empty/garbage queries -> [].

### B. Retrieval integration (retrieve.query)

1. **Recall**: candidates = vector ∪ graph ∪ BM25-top-40. The lexical
   channel can now SURFACE candidates, not just rescore them.
2. **c_lexical upgrade**: `max(substring_fraction, bm25_rank_score)` where
   `bm25_rank_score = 1/(1 + rank_position)` (top BM25 hit ~1.0, decaying)
   -- an RRF-style rank fusion folded into the existing zeta component, so
   the 6-component schema + auto-tuner stay intact.
3. **Exact-name finisher**: when a candidate's `name` (len >= 4,
   case-insensitive) appears verbatim in the prompt, multiply the final
   score by `cfg.exact_name_boost` (default 1.25) -- same multiplicative
   pattern as the isolation penalty. Asking for a thing BY NAME is the
   strongest intent signal retrieval gets.

### C. NULL-key data fix (ingest)

NULL-keyed foreign nodes (e.g. the aibox AGENTS.md docs node) are
cross-cutting by the v1.2.1 contract and leak into every scoped query.
Root cause: directory sources registered without a `project_key`. Fix at
reindex time: for sources of kind code_repo / docs_dir / plan_dir /
memory_dir with `project_key IS NULL`, derive it from the source root
(`paths.resolve_project_key`), persist on the source row, and backfill
owned nodes (`UPDATE nodes SET project_key=? WHERE project_key IS NULL AND
source_path LIKE root || '%'`). `claude_md` sources stay None -- global
memory is cross-cutting BY DESIGN.

## 3. Anti-goals

- No new scoring WEIGHTS (no 7th component; the tuner contract is stable).
- No embedding/model changes; no new deps (FTS5 ships in CPython's sqlite).
- `mnemo eval` stays a report, not a CI gate.

## 4. DoD

1. `mnemo eval` on the live corpus STRICTLY beats hit@5=0.21 (goal >= 0.43,
   i.e. 6/14); numbers recorded in the design doc + handover.
2. Seeded unit proof: a code node literally named like the query term
   outranks a long prose node that merely paraphrases it.
3. NULL-key backfill verified live: the aibox docs node carries
   D--Repository-aibox-prod-all after one reindex.
4. Full suite green (ranking-order tests updated honestly where the new
   scoring legitimately changes order); ruff clean.

## 5. Build order (TDD; one commit per step)

1. Store: nodes_fts schema + sync + backfill-on-create + bm25_search +
   rebuild_fts.
2. Retrieve: BM25 candidates + c_lexical fusion + exact-name finisher.
3. Ingest: NULL-key derivation + node backfill.
4. Live: reindex (applies backfill) + `mnemo eval` re-run -> record.
5. Bump 5.26.0 -> 5.27.0 + CHANGELOG + full verify + ship.

## 6. BUILD NOTE -- results (2026-06-10)

Live `mnemo eval` on the 18,073-node corpus, full stack (BM25 recall +
RRF/zeta fusion + exact-name finisher + multi-key auto-scope + NULL-key
backfill):

| metric | v5.26.0 baseline | v5.27.0 | delta |
|--------|------------------|---------|-------|
| hit@1  | 0.14 | **0.21** | +50% |
| hit@5  | 0.21 | **0.36** | +71% |
| MRR    | 0.16 | **0.25** | +56% |

DoD #1 (strictly beat hit@5=0.21) -- MET. The stretch 0.43 was not
reached (5/14, not 6/14). NULL-keyed nodes dropped 2289 -> 353 (the 353
are legitimately cross-cutting: claude_md globals + keyless rows).

The 3 remaining misses are honest, deferred to v5.28.0:
- **snake_case vs spaced tokens**: "post tool use hook decide a file is
  memory shaped" doesn't match `_is_memory_shaped` (the exact-name
  finisher needs the verbatim token; BM25 splits on `_`). v5.28.0:
  snake_case-aware tokenization + identifier expansion.
- **code outranked by term-dense prose**: doctor.py's
  `check_plugin_registered` loses to handovers that discuss "doctor" +
  "plugin registered" heavily. v5.28.0: a code-node type boost for
  how/where questions.
- **corpus coverage**: install.sh / install.ps1 live at the repo root and
  are not inside any indexed source -- unwinnable for retrieval until the
  repo-root scripts are indexed. Not a ranking bug.

FTS dup-on-concurrent-open: the first "backfill if empty" guard
double-inserted when the daemon + a CLI reindex opened a fresh index at
once (36022 rows for 18073 nodes). Fixed to self-heal by rebuilding when
`count(nodes_fts) != count(nodes)` -- a no-op COUNT compare in the steady
state, idempotent across concurrent opens.
