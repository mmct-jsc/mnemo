## Summary

**v2.1.1 -- Nebula UX polish + scaling palette.** Seven follow-ups
on top of v2.1.0 that landed on the release branch after the
v2.1.0 tag. Common thread: previously-implicit per-type behavior
was made explicit and palette-driven so the UI absorbs new node
types without per-file edits.

### Headline: single-source node-type palette

Before, the same per-category palette was duplicated across three
places: CSS rules (`.badge.type-X`, `.bar-fill.type-fill-X`,
`.swatch-X`, `.ntype-X`), a JS `TYPE_COLORS` dict in `graph.html`,
and ad-hoc inline palettes in a couple of templates. When v2.0
added 7 code_* types, only some of the three got updated -- the
dashboard "Memory by type" panel + Nebula filter chips silently
fell back to neutral (visible symptom: all-blue chips + invisible
progress bars).

**Now:** `daemon/mnemo/ui/palette.py` owns the dict. Exposed to
Jinja via `templates.env.globals["type_colors"]` and to JS via
`window.MNEMO_TYPE_COLORS`. Templates stamp
`style="--type-color: <hex>"` inline; CSS has ONE generic rule per
class family reading `var(--type-color, fallback)`. Adding a new
node type from now on is one line in `palette.py`.

### Type-aware body preview

A single helper -- `window.mnemoRenderBody(targetEl, body, { type,
sourcePath })` -- branches on three pipelines: `code_*` ->
Prism-highlighted `<pre><code>`; `commit` -> escaped plain `<pre>`;
everything else -> marked + DOMPurify markdown. The node detail
Preview tab, search-result popover, and Nebula side panel all
share this one helper.

### Other fixes

* **Empty-canvas tap on Nebula now deselects.** Guard switched from
  `evt.target === cy` (fails in minified Cytoscape builds because
  the core is wrapped in an obfuscated class) to a capability
  test (`typeof t.isNode !== 'function'`).
* **Force-layout snapshot restore was dead code** -- guard checked
  the wrong layout name. Switching to rings/tree and back now
  returns to exact baseline positions (max drift 0 px over 422
  nodes).
* **Two-arrow bug in Nebula file tree** -- a redundant CSS-content
  `::before { content: "▸" }` was painting a second chevron;
  Chrome 128+ also reserves a phantom flex slot for `<details>`
  disclosure widgets inside any `<summary>` with `display: flex`.
  Cure: removed the duplicate rule + restructured summary children
  into an inner `display:flex` wrapper.
* **Connections count rendered as literal text** -- template was
  hand-escaping Alpine mustache through Jinja2, but Alpine doesn't
  use mustache for text interpolation. Replaced with
  `<span x-text="neighbors.length">`.
* **Force-layout ran twice on first load** -- Alpine auto-invokes
  any method named `init()` on the `x-data` object; the redundant
  `x-init="init()"` paired with `x-data` ran init() twice. Dropped
  the attribute in five templates.
* **Native `<details>` marker** -- per spec, `::marker` only accepts
  color/content/font-*/white-space/text-* properties. `display:
  none` is silently ignored. Switched to allowed levers
  (`content: ""; font-size: 0; color: transparent`) plus CSS
  cache-busting via `?v={{ mnemo_version }}` on `/static/app.css`.

### Branch naming

This release lives on `release/2.1.1` to honor the
**one-branch-per-minor-version** policy captured in memory after
the v2.0+v2.1 stacking mistake. The same content was previously
pushed to `release/2.0.0`; you can delete that branch after merge.
