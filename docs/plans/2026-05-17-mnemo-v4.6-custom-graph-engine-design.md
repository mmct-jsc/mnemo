# mnemo v4.6.0 — Custom Graph Engine ("Nebula", rebuilt) — Design

**Status:** DESIGN (validate before implementation)
**Date:** 2026-05-17
**Branch:** `release/4.6.0` (off `main` 5d3f5c5 = v4.5.3)
**Supersedes:** the cosmos.gl → sigma.js renderer-swap arc (v4.5.0–v4.5.3). All four
shipped iterations were rejected on live review (mess / lag / weird layout / ugly
atmosphere frame). This is the architecture-question gate, well past the
3-failed-fixes rule: stop patching a third-party renderer, build a purpose-built
stack.

---

## 1. Why (root cause, evidence-based)

Two **independent** problems were repeatedly treated as one. Both confirmed by a
code audit of the shipped v4.5.3.

### 1a. A layout-algorithm problem (the "blob" + "confetti")

`mnemo/ui/graph_layout.py` uses plain Fruchterman–Reingold with three fatal
properties:

- **Repulsion is k-nearest-neighbour only** (`_KNN = 16`, via `cKDTree`). Distant
  communities exert **zero** mutual repulsion, so nothing pushes clusters apart.
- **Undifferentiated centroid gravity** (`(centroid - gpos) * 0.004 * …`) with no
  community-separation term — the system equilibrates into one centrally-weighted
  mass.
- **p95-radius normalize** (`gpos *= _R_CORE / r95`) forces the 95th-percentile
  radius to exactly `_R_CORE`; on an 11k giant the bulk packs into a few hundred
  px → a featureless dense disk by construction.
- The 2,298 small components/singletons get **one golden-angle dot each** on a
  thin annulus (`rr = _R_CORE*(1.32+0.92·√frac)`) → sparse confetti, not a halo.

Every other visual defect is a **symptom of the blob**: 15.5k edges at 0.34 alpha
wash into the cyan fill because their endpoints are not spatially separated; hubs
merge into giant flat discs because the core is collapsed. **No renderer can make
a blob look good.**

### 1b. A self-inflicted CSS-overlay problem (the white frame + washed bg)

The v4.5.3 "atmosphere" is a **transparent** WebGL canvas (`#cy-nebula
{background:transparent}`) with `.nebula-canvas::after` a near-white starfield
(`rgba(214,222,255,0.9)` dots) at `inset:-10%` and `::before` purple/cyan blooms
at `inset:-14%`. The starfield tiles past the canvas edge → an ugly fuzzy white
**frame**; the blooms + stars show **through and over** the nodes → the washed
lavender background. This is decorative chrome fighting the data — and it has been
rejected on every iteration.

### 1c. The performance problem

The v4.5.x lag was a perpetual rAF loop doing `cam.setState()` + a full
`sig.refresh()` (11k points + 15.5k curved edges) every frame. v4.5.3 deleted the
loop (correct) but sigma's curved-edge program + software-WebGL still make 11k
heavy, and on-demand re-render of sigma geometry is not free.

## 2. What the field does (research, "reference from other libraries")

Convergent findings from a study of cosmos.gl/Cosmograph, Sigma v3, Graphistry,
Gephi ForceAtlas2/LinLog, d3-force, ngraph, SFDP:

1. **The de-blob fix is a known algorithm**, not tuning: **ForceAtlas2 with
   LinLog mode + outbound-attraction-distribution + low (non-strong) gravity +
   Barnes-Hut quadtree repulsion**. LinLog attraction `F = log(1+d)` (vs FR's
   `F = d`) makes clusters *tight and visibly separated*. Outbound attraction
   (divide edge attraction by source mass) pushes hubs to the periphery instead
   of piling them in the centre. This is what Gephi/Graphistry use to turn a
   hairball into readable communities.
2. **GPU layout is not worth it at 11k.** cosmos.gl's ping-pong-float-texture +
   quadtree-in-texture machinery pays off above ~100k nodes. At 11k a correct
   CPU/numpy Barnes-Hut layout runs in well under a second and we already cache
   it. **The GPU belongs to rendering, not layout, at our scale.**
3. **Crisp + fast rendering is a small set of standard techniques**: instanced
   point-sprite nodes (one draw call), a **signed-distance-field circle** in the
   fragment shader with `fwidth` AA (crisp at *any* zoom), additive-blended soft
   glow, **low-alpha straight edges so overlap density reveals structure**
   (Graphistry's key trick), a true `gl.clearColor` dark theme (no CSS overlay),
   render-only-when-dirty, and label/element LOD.
4. **regl** ships a single pinned UMD file (`dist/regl.min.js`, no Node build) —
   it fits mnemo's vendor model exactly (the way sigma was vendored).

## 3. Goals / Non-goals

**Goals**

- Replace sigma.js + graphology + the CSS atmosphere with a purpose-built stack:
  two custom "libraries" (the user's framing).
  - **G1 — Layout engine** (server-side Python, numpy/scipy): a high-quality,
    deterministic, cached layout that *separates communities* (no blob, no
    confetti).
  - **G2 — Renderer** (client-side, single vendored regl-based JS, no Node
    build): crisp, fast, fully interactive, dark-themed.
- Optimize **both**: smooth zoom/pan/drag/click at 11k nodes / 15.5k edges, idle
  = zero cost; separated communities, crisp nodes, readable edges, a dark theme
  that *matches the existing C1 palette*, no decorative frame.
- One **v4.6.0** release. Nothing ships until verified end-to-end on the real
  ~11,057-node / ~15,500-edge scope.
- A complete, validated **design doc (this) before implementation**, then a TDD
  execution plan.

**Non-goals (YAGNI)**

- GPU-based layout simulation (cosmos-style) — overkill + high risk at 11k.
- Any decorative CSS atmosphere / animated starfield / twinkle / pulse — the
  repeatedly-rejected failure source. Deleted, not re-skinned.
- Force-directed edge bundling, curved edges — last resort only; start with
  low-alpha straight edges (the proven readable default).
- 3D, time animation, multi-graph, a general-purpose public graph library. This
  renderer is purpose-built for mnemo's one view.
- Keeping any sigma / graphology / cosmos.gl code or vendor file.

## 4. Visual identity (decided)

"Most suitable + match current theme & mindset": **clean, crisp, professional,
dark — Gephi/Graphistry-grade.** Concretely:

- True opaque dark canvas via `gl.clearColor`, matched to C1 `--bg`
  (`#07090f`-ish). The canvas is **opaque** — nothing can ever bleed over the
  nodes again.
- Nodes: crisp SDF circles, per-type C1 palette colours (the existing
  `palette.py` / `--type-color` system), a thin slightly-darker border ring so
  adjacent nodes stay distinct, and a **restrained additive glow** — this is the
  only flourish, and it gives a tasteful "nebula" feel *without any CSS*.
- Edges: thin, low-alpha (~0.06–0.15) in the C1 dim colour; dense regions sum
  into soft "highways" that *reveal* structure. A selected/hovered node's
  incident edges brighten to `--accent` (#7ee7e0) and draw on top.
- No animation except: camera ease on click-to-centre, and the highlight
  fade. No idle motion. The graph is the aesthetic; the data is the art.

This preserves the established dark + teal-accent C1 identity and the "Nebula"
name, drops every element that has been rejected.

## 5. Part A — Layout engine (`mnemo/ui/graph_layout.py`, full rewrite)

A deterministic **ForceAtlas2 / LinLog** layout, server-side, numpy/scipy, cached
exactly as today (the v4.5.1 server-side architecture was accepted; only the
*algorithm* and *normalize* were wrong).

**Algorithm**

- **Barnes-Hut quadtree repulsion**, θ ≈ 0.7 — O(n log n) and, crucially, *all
  nodes repel all regions* (not kNN-16). This single change is what lets distant
  communities push apart. Implemented as a vectorised numpy quadtree (build once
  per iteration; aggregate mass + centroid per cell; traverse coarse cells for
  distant groups).
- **LinLog attraction**: `F_a = log(1 + d)` along edges (replaces FR `F_a = d²/k`)
  — tight, *separated* clusters.
- **Outbound attraction distribution**: divide each edge's attraction by the
  source node's mass (`deg + 1`) — hubs go to the periphery, not the core.
- **Edge-weight influence**: mnemo edges carry a confidence weight — fold it into
  attraction (`weight^edgeWeightInfluence`, influence ≈ 1).
- **Gravity**: weak, **non-strong** FA2 gravity (`F_g = gravity · mass ·
  dir_to_centre`, gravity tuned ~0.5–1.0, strongGravity OFF) — just enough to
  keep the drawing compact and singletons from drifting; never enough to crush
  the core.
- **Adaptive convergence**: FA2 global speed with per-node swing/traction and a
  max-step clamp; fixed iteration cap (e.g. 300–500) with early-exit on low total
  swing. Deterministic seeded init (small random or a spectral/BFS seed).
- **Normalize by full extent**, not p95: centre at origin, scale so the **full
  bounding box** fits a stable world rectangle. (The p95-normalize was a primary
  blob cause.)
- **Small components + singletons**: not one-dot-per-component. Lay out each small
  component with the same FA2 (tiny, instant). Then **deterministically pack** all
  small components + singletons into a *dense, textured* band hugging the giant —
  e.g. grid/quadtree bin-packing of component bounding boxes with light jitter, or
  dense concentric clusters sized by component — so they read as a populated halo,
  not sparse confetti. Singletons cluster into a dense jittered field, never flung
  individually onto a wide ring.

**Determinism & cache**: fixed seed → byte-identical output; existing
`scope_key + fingerprint + LAYOUT_VERSION` cache + `_ensure_layout_async`
background-thread compute kept verbatim. `LAYOUT_VERSION` → `fa2-linlog-1`
(busts the cache once; recomputes the better layout). Output contract unchanged:
flat `[x0,y0,x1,y1,…]` in node order; the client applies index-aligned. Compute
is offline + cached, so even multi-second convergence is paid once per scope.

## 6. Part B — Renderer (`mnemo/ui/static/vendor/`)

Vendor `regl.min.js` (pinned UMD, no Node build) + author **`nebula-gl.js`**, our
own small purpose-built renderer (plain IIFE, vendored alongside). Public surface
is intentionally tiny and mnemo-shaped:

```
NebulaGL.create(canvasEl, { nodes, edges, theme })  ->  handle
handle.setHighlight(idSet | null)   // dim others, brighten set + incident edges
handle.select(id | null)            // centre + emphasise (camera ease)
handle.hover(id | null)
handle.on('clickNode' | 'clickStage' | 'hoverNode', cb)
handle.fit()  /  handle.camera     // pan/zoom state
handle.destroy()
```

**Rendering**

- **Nodes**: one instanced draw call for all ~11k. Fragment shader = SDF circle
  with `fwidth`-based AA → crisp at any zoom (fixes the giant blurry discs). Size
  = a sane clamped function of degree, **modulated by camera zoom** so a node is
  never a screen-filling disc. Thin darker SDF border ring. A second, larger,
  faint **additive-blend** sprite per node = the restrained glow.
- **Edges**: one instanced draw call; thin straight quads (or `GL_LINES`) at low
  alpha. Highlighted/selected incident edges in a second pass, brighter,
  `--accent`, on top.
- **Theme**: `gl.clearColor` = C1 dark base; colours from the existing
  `palette.py` per-type system passed in `theme`. **No CSS atmosphere at all** —
  `.nebula-canvas::before/::after` and the `@keyframes` are deleted; the canvas
  is opaque.
- **Camera**: renderer owns pan (drag), zoom-to-cursor (wheel), node-drag, and a
  short eased click-to-centre. A single rAF that **renders only when dirty**
  (camera/selection/hover/drag changed) — idle = zero GPU/CPU (the definitive lag
  fix, now in *our* code, nothing to fight).
- **Picking**: CPU hit-test via a spatial index (a quadtree built once from the
  served positions) for click/hover — O(log n), zero per-frame cost. (GPU
  colour-picking is the fallback if CPU proves insufficient.)
- **LOD**: labels only for selected / hovered / neighbours (and optionally above
  a zoom threshold), drawn on a thin 2D-canvas overlay (the dark themed pill, as
  today's `nbDrawLabel`, but renderer-driven). Optional edge-skip while actively
  dragging the camera if measurement shows it's needed (likely not, with
  low-alpha edges + dirty-only rendering).

## 7. Part C — Integration & contracts

- **`graph.html`**: delete all sigma/graphology code (`renderCanvas`, reducers,
  `_startLife`/`_positionPulse`, vendor `<script>`s). Keep: the server-layout
  poll (`_awaitServerLayout`, scope/fingerprint), the side-panel / neighbours /
  citation Alpine contract, the `mnemo-highlight-nodes` / `mnemo-select-node`
  document listeners (now call `handle.setHighlight` / `handle.select` — the
  gotcha-31 / C3-honesty loop stays closed: the graph itself still highlights),
  the C1 responsive shell, the loading/booting states.
- **`app.css`**: delete `.nebula-canvas::before/::after`, `@keyframes
  nebula-drift/twinkle/parallax`, `.nebula-pulse-*`. `.nebula-canvas` → a plain
  opaque dark container; the mount hosts the `<canvas>` (opaque). Keep the
  responsive shell rules.
- **`routes.py`**: `LAYOUT_VERSION` → `fa2-linlog-1` (one-time cache bust +
  recompute).
- **Vendor**: add pinned `regl.min.js` + `nebula-gl.js`; remove `sigma.min.js`,
  `graphology.umd.min.js`. No CDN, no Node build (stack rule).
- **Tests**:
  - `test_nebula_renderer.py` → evolve the TOMBSTONE: cosmos **and** sigma **and**
    graphology tokens all forbidden; assert `regl` + `nebula-gl` vendored + loaded;
    assert NO `@keyframes nebula-*` / no `.nebula-canvas::after` atmosphere;
    assert the highlight/select/hover/drag/cite surface present; the C1 shell
    intact.
  - `test_graph_layout_server.py` → assert FA2/LinLog quality **numerically**
    (this is the real gate, no GPU needed): deterministic, all finite, bounded;
    **community separation** — mean edge length ≪ mean random-pair length AND the
    node-density distribution is *multi-modal / structured* (e.g. a coarse grid
    occupancy or nearest-neighbour-distance histogram shows separated clusters,
    not a single dense disk); small components packed **densely** (high local
    occupancy) not on a sparse thin ring.

## 8. Verification strategy (given the documented headless limit)

The headless preview is software-WebGL in a hidden 0-viewport tab → screenshots
and heavy evals time out; rAF/visibility no-ops. v4.5.x kept hitting this. This
design **moves the main quality gate off the GPU**:

1. **Layout quality is now numerically unit-tested in pytest** — the actual
   failure (the blob) is a property of the Python output and is asserted with
   metrics (separation, multi-modality, packing density) on a real-shaped
   synthetic graph **and** spot-checked on the real 11,057-node cached output via
   the layout endpoint. No GPU required to prove the #1 problem is fixed.
2. **Served-asset probes** (authoritative): regl + nebula-gl present; sigma /
   graphology / cosmos / CSS-atmosphere absent.
3. **Renderer smoke via eval**: the module loads, exposes its API, and
   `NebulaGL.create` on a tiny 2-node graph returns without throwing (light, well
   under the eval cap).
4. **The user's focused GPU browser is the final visual arbiter** — stated
   explicitly and unavoidably; but unlike v4.5.x, the dominant failure is
   provable without it.

## 9. Risks & mitigations

| Risk | Mitigation |
|---|---|
| regl single-file UMD availability | Confirmed: regl ships `dist/regl.min.js`. Fallback: raw WebGL2 or twgl (more code). |
| numpy Barnes-Hut correctness/perf | Well-known algorithm; vectorised; iteration-capped; offline + cached so even ~10 s once is fine. Numeric tests gate quality. |
| Big-bang scope (user chose one chapter) | Strict internal TDD **phases**, each committed: (1) layout engine + numeric quality gates, (2) renderer + smoke, (3) integration + contract tests, (4) full verify. One release at the end. |
| Headless WebGL unobservable | The #1 failure (layout) is now pytest-provable; renderer is structurally simple + user-GPU-verified. |
| Highlight/citation regression | The document-listener + side-panel contract is preserved verbatim; covered by evolved contract tests. |

## 11. Open questions for design validation

1. Small-component/singleton packing: dense grid-pack of component bboxes around
   the giant **(recommended)** vs. dense concentric clusters sized by component?
   (Both are deterministic + bounded; grid-pack is simpler + denser.)
2. Labels: 2D-canvas overlay (crisp text, simplest) **(recommended)** vs. SDF
   text in GL (heavier, unnecessary at our label counts)?
3. Is "clean professional dark + restrained additive glow" the final aesthetic
   call, or should the glow be omitted entirely for maximum austerity?

Everything else is considered settled by the research + the user's two answers
(clean/dark/C1-matched; one v4.6.0 chapter; design doc first).
