# mnemo v1.1 — Beyond Claude Code

**Status:** approved 2026-05-10
**Codename:** Beyond Claude Code (multi-IDE / multi-AI)
**Predecessor:** v1.0.5

## 1. Goals, non-goals, scope

### Goal

Make mnemo usable from any IDE or any program that calls an LLM SDK,
while preserving the local-first, token-budgeted, citation-back contract.

### What ships in v1.1

1. A versioned public HTTP protocol under `/v1/...` with a published
   OpenAPI spec.
2. A VS Code marketplace extension (`mnemo-vscode`) — palette commands,
   sidebar TreeView, status bar item, `@mnemo` chat participant.
3. A PyPI package (`mnemo-middleware`) — `retrieve_context()` helper
   plus a `patch(client)` monkey-patcher for OpenAI, Anthropic, Google,
   and Ollama clients.
4. Broader file ingest (`.md`, `.txt`, `.pdf`) plus per-source
   include/exclude glob patterns.
5. A Sources page in the UI that lets users add, edit, remove, and
   toggle sources without touching the CLI.
6. Three new workflow skills: `mnemo:plan`, `mnemo:retro`,
   `mnemo:incident`.

### Non-goals

- No cloud sync, no multi-user, no auth. Local-first stays the contract.
- No JetBrains plugin (defer to v1.2 or v2).
- No code-graph parsing — that is v2's job.
- No chat surface — that is v3's job.
- No SaaS-side ingestion (Notion, Slack, Confluence, GitHub Issues) —
  that is v1.2's job.
- No `mnemo:doc` skill. Deferred to v3 alongside the chat companion,
  where doc-context UX has a natural home.
- No automatic VS Code marketplace publish in v1.1. The `.vsix` is
  attached to the GitHub release; marketplace publishing is a v1.2
  decision.

### Success criteria

- A developer can drop two lines (`import mnemo_middleware as mm;
  mm.patch(client)`) into a script using the OpenAI SDK and have the
  next chat completion auto-receive memory citations.
- A developer who installs the VS Code extension can type `@mnemo` in
  any chat-participant-aware extension (Copilot Chat, Cody) and see
  budget-capped retrieval inline.
- The existing Claude Code plugin keeps working unchanged.
- Old API paths (`/query`, `/nodes`, `/sources`, `/reindex`, `/config`)
  return `308` to their `/v1/...` equivalent for the entire v1.1
  series; they are removed in v1.2.

### Invariants

- One daemon per machine. All clients converge on `127.0.0.1:7373`.
  No client embeds the model or duplicates the store.
- No client-to-client coupling. Clients only talk to the daemon.
- Existing daemon code stays the engine. v1.1 daemon work is purely
  additive: new endpoints, new source columns, new parsers. The
  retrieval pipeline does not change.
- Adapters are thin (target ≤ 600 LOC each). Heavy lifting stays in
  the daemon.

## 3. Public protocol surface

### URL versioning

All public endpoints live under `/v1/`. Each response carries
`X-Mnemo-Api-Version: 1`. Legacy paths (`/query`, `/nodes`, `/sources`,
`/reindex`, `/config`, `/projects`, `/intents`) return `308` to their
`/v1/...` equivalent for the v1.1 series and are removed in v1.2. UI
HTML routes (`/`, `/nodes-page`, `/audit-page`, `/sources-page`,
`/settings`, `/graph`, `/node/<id>`) are unchanged — they are for
browsers, not adapters.

### OpenAPI

FastAPI builds a schema from route signatures. v1.1 makes it useful:

1. Tag every public route with `tags=["v1"]`.
2. Mark internal routes (UI HTML, HTMX fragments, static assets) with
   `include_in_schema=False`.
3. Expose the v1-only schema at `GET /v1/openapi.json`.

The VS Code extension uses `openapi-typescript` at build time to
generate request and response types from the spec. The middleware uses
`httpx` plus dataclasses derived from the same schema.

### New endpoints

| Method + path | Purpose |
|---|---|
| `POST /v1/projects/resolve` | `{path}` → `{project_key}`. Canonical project-key derivation. |
| `GET /v1/projects/active` | Returns `{project_key, path, since}` or `null`. |
| `POST /v1/projects/active` | `{path}` → daemon resolves and persists. Returns active record. |
| `DELETE /v1/projects/active` | Clears the active project. |
| `POST /v1/sources` | Create source: `{path, kind, include?, exclude?, project_key?}`. |
| `PATCH /v1/sources/{id}` | Update enabled flag, include, exclude, project_key. |
| `DELETE /v1/sources/{id}` | Remove a source. |
| `POST /v1/query` | Existing, but now accepts optional `project_key`. Falls back to active project. |

### Project-key derivation algorithm

Canonical algorithm; both spec and daemon implement it. CI compares
adapter ports against the daemon's `/v1/projects/resolve` output to
catch drift.

```
project_key(path):
  1. Resolve symlinks; absolute path; case-sensitive on POSIX,
     lowercase drive letter on Windows.
  2. Replace ":" with "-"; replace "/" and "\\" with "-".
  3. Collapse consecutive "-" runs.
  4. Strip leading and trailing "-".
```

A fixture file `daemon/tests/fixtures/project_keys.json` lists 40+
`(path, expected_key)` pairs covering Linux, macOS, and Windows. Every
adapter test suite POSTs each input to `/v1/projects/resolve` and
asserts equivalence with its local implementation. Drift fails CI.

## 4. Daemon changes

Five additive changes.

### 4.1 Active-project state

- New table:

  ```sql
  CREATE TABLE active_project (
    singleton_id INTEGER PRIMARY KEY CHECK (singleton_id = 1),
    project_key  TEXT,
    path         TEXT,
    since        INTEGER
  );
  ```

- `Store.get_active_project()`, `Store.set_active_project(path)`,
  `Store.clear_active_project()`.
- `retrieve.query()` gets a new `project_key` argument; falls back to
  the active project when absent (per the hybrid contract).

### 4.2 Source filters

- Migration adds `include TEXT, exclude TEXT` to the `sources` table.
  Both store comma-joined glob patterns. Defaults:
  - memory dirs: include = `*.md, *.txt, *.pdf`
  - claude_md: include = `**/*.md`
  - explicit single files: empty patterns
- `ingest.discover()` walks each source and applies a `pathspec.PathSpec`
  (gitignore-style matcher) for include + exclude.
- Existing sources back-fill to the new defaults on first run.

### 4.3 New file-format parsers

```
ingest/
  parsers/
    __init__.py     (registry: extension -> parser)
    md.py           (existing logic, extracted)
    txt.py          (NEW)
    pdf.py          (NEW)
```

A `Parser` protocol:

```python
class Parser(Protocol):
    def supports(self, suffix: str) -> bool: ...
    def parse(self, path: Path) -> tuple[Frontmatter, str]: ...
```

`txt.py`: reads the file, strips trailing whitespace, treats it as a
single chunk.

`pdf.py`: uses `pypdf` (small, pure-Python). Extracts per-page text;
chunks at 1500 chars with 200-char overlap. The same chunker is shared
with `txt.py`.

A `parse_quality: 'high'|'low'` field on parsed nodes is added to the
source row. Low-quality (PDF) nodes can be deprioritized in retrieval
without re-parsing.

### 4.4 Project-key resolver

`paths.project_key(path: Path) -> str` implements the canonical
algorithm. The existing `paths.project_key_from_dir` is updated to
call this; old behavior preserved for callers that already use it.

### 4.5 Versioning + redirect middleware

All public routes get `tags=["v1"]` and move under `/v1/...`. Internal
UI/HTMX routes keep `include_in_schema=False`.

A small middleware: if the request path starts with one of the legacy
roots **and** `Accept` includes JSON (so we don't break HTML pages),
respond `308 Permanent Redirect` with `Location: /v1{path}`.

## 5. VS Code extension

**Repo:** `mmct-jsc/mnemo-vscode` (separate, see Section 9).
**Stack:** TypeScript, esbuild, vsce.

### Surfaces

1. **Status bar item** (right side, low priority):
   `$(brain) mnemo · D--Repo-name · 17ms`. Shows daemon up/down,
   active project, last query latency. Click → palette with Activate /
   Switch Project / Open UI / Stop daemon.

2. **Command palette commands**:
   - `mnemo: Query` — input box, opens result in a side-panel webview
     with hits and citations. Selected text in the editor is the
     default query.
   - `mnemo: Add Note` — webview form (description, body, type,
     project) → `POST /v1/nodes`.
   - `mnemo: Set Active Project` — workspace folder by default; falls
     back to a folder picker.
   - `mnemo: Open UI` — `vscode.env.openExternal('http://127.0.0.1:7373')`.
   - `mnemo: Reindex` — `POST /v1/reindex`, toast on completion.

3. **Sidebar TreeView** (`mnemo` activity bar icon):
   - Active project (header, click to switch).
   - Recent queries (last 10, pulled from `/v1/audit` on focus). Click
     expands to the hit list; click a hit to open the node detail in
     the side panel.
   - Pinned nodes (stored in extension state; right-click any node →
     Pin).

4. **Chat participant** — `vscode.chat.registerChatParticipant('mnemo')`.
   Typing `@mnemo what's our MQTT auth pattern?` in any
   chat-participant-aware extension (Copilot Chat, Cody) calls our
   handler:
   - `POST /v1/query` with the chat prompt, active project, default
     budget 800.
   - Streams formatted hits as **chat references** so the user sees
     clickable `[mnemo:abc123]` chips alongside the assistant's reply.
   - Slash sub-commands: `@mnemo /recall <q>`, `@mnemo /add`,
     `@mnemo /sources`.

### Settings

| Key | Default |
|---|---|
| `mnemo.daemonUrl` | `http://127.0.0.1:7373` |
| `mnemo.budgetTokens` | `800` |
| `mnemo.autoActivate` | `true` (sets active project to the workspace folder on extension activation) |

### Distribution

CI: lint (eslint), build (esbuild), `vsce package`, attach `.vsix` to
a GitHub release on `v*.*.*` tag. **No marketplace publish in v1.1.**
Users install via `code --install-extension mnemo-vscode-1.1.0.vsix`
or by downloading the `.vsix` from the release page.

## 6. SDK middleware

**Package:** `mnemo-middleware` on PyPI.
**Lives in:** `clients/middleware-py/` of the main `mnemo` repo
(separate `pyproject.toml`).
**Dependencies:** only `httpx`.

### Public API

```python
import mnemo_middleware as mm

# 1. Manual: get context, do whatever with it
ctx = mm.retrieve_context(
    "how do we handle MQTT auth?",
    project_key=None,    # optional; daemon uses active project if None
    k=5,
    budget_tokens=800,
)

# 2. Opt-in patcher: monkey-patches a known SDK client
from openai import OpenAI
client = OpenAI()
mm.patch(client, mode='auto')          # default
mm.patch(client, mode='once')          # inject once on first call only
mm.patch(client, mode='every')         # inject every call (one-shot apps)

# 3. Reverse the patch
mm.unpatch(client)
```

### Injection modes

`mode='auto'` (default) re-injects only when one of these fires:

1. `messages` has length ≤ 2 (first or near-first turn).
2. `messages[0]` differs from the previously seen call (new
   conversation).
3. The user's last message scores below a similarity threshold against
   the previous mnemo block (topic shift detected — uses the daemon's
   embedder via `embed_text`).

The patcher keeps a small per-client state dict:
`{ "last_messages_hash", "last_query_embedding", "last_mnemo_block" }`.
Memory cost: a few KB per patched client.

`mode='every'` injects on every call. Use for one-shot evaluators that
make 1000 independent prompts.

`mode='once'` injects only on the very first call after `patch()`.
Use for persistent agents that should establish context once and run.

### Provider shims

One file per provider in `_shims/`: `openai.py`, `anthropic.py`,
`google.py`, `ollama.py`. Each shim:

- Detects the client class by `type(client).__module__`.
- Wraps the relevant `create` / `messages.create` method.
- Pulls the user's last message (or all user messages joined) as the
  query.
- Calls `retrieve_context()`.
- Prepends or merges a system message:
  `f"[Project memory — cite as shown]\n\n{ctx}"`.
- Calls the original method, returns its result.

Unsupported client types raise `mm.UnsupportedClient`.

### Prompt-caching integration

When `mode != 'every'` AND the mnemo block is reused across calls,
shims emit provider-specific cache hints:

- **Anthropic**: `cache_control: { type: 'ephemeral' }` on the system
  block when block ≥ 1024 tokens. 90% discount on cache hits, 5-min
  TTL.
- **OpenAI**: places the mnemo block at the very top of the messages
  array so the cache prefix is stable. OpenAI auto-caches matching
  prefixes ≥ 1024 tokens.
- **Google / Ollama**: no formal caching, but token usage stays
  bounded because we replace, not append, the system block.

### Configuration (env vars)

| Env var | Default |
|---|---|
| `MNEMO_DAEMON_URL` | `http://127.0.0.1:7373` |
| `MNEMO_DEFAULT_BUDGET` | `800` |
| `MNEMO_TIMEOUT` | `2.0` (seconds) |

### Failure modes

A slow or unavailable daemon should never block a model call.

- Daemon down → `retrieve_context()` returns `""`, patcher proceeds
  without injection. Logged at WARNING.
- Daemon timeout → same.
- Patcher exception → caught, logged, original method called
  unchanged.

The middleware is additive. Model calls always succeed even when
mnemo is unavailable. Non-negotiable.

### Tests

`pytest` against an `httpx.MockTransport` for unit tests. CI runs one
integration test against a live daemon (`uv run mnemo daemon start`)
that hits real `/v1/query`.

## 7. New workflow skills

Three skills, all under `skills/`. Each gets a slash command at
`commands/`. They are markdown files — usable in Claude Code today
and in any future IDE that adopts a similar skills protocol.

### 7.1 `mnemo:plan` (rigid, 6 phases)

Closes the gap between "I have an idea" and `mnemo:implement-platform`.
Output: a design doc the user reviews before any code.

| Phase | What |
|---|---|
| 1. Pull context | `mnemo query` on the user's request, with `intent_tags=['plan']` to bias toward project + plan_doc nodes. Show top hits with citations. |
| 2. Brainstorm | Ask refining questions one at a time. |
| 3. Approaches | Propose 2-3 architectural approaches; recommend one; justify. |
| 4. Decisions | One architecture decision per round, user-approved. |
| 5. Write | Render the validated design to `docs/plans/<date>-<topic>-design.md`. Cite every borrowed pattern with `[mnemo:<id>]`. |
| 6. Done-criteria | Append a phased roadmap, one commit per phase. Stop. The user runs `/mnemo-implement-platform` next. |

### 7.2 `mnemo:retro` (flexible, 4 phases)

End-of-session lesson extraction.

| Phase | What |
|---|---|
| 1. Sweep | Pull recent audit log (last N hours), recent file edits (`git log --since`), and any new `docs/plans/*` artifacts. |
| 2. Propose | Surface 0-N candidate memory entries: lessons, gotchas, durable techniques, decisions. Each candidate has a `type`, draft `name`, draft `description`, draft body, and a confidence score. |
| 3. Triage | User accepts, edits, or rejects each. Rejected ones are logged so we don't propose the same thing tomorrow. |
| 4. Write | Accepted entries become files under the active project's memory dir. `mnemo reindex` runs. |

### 7.3 `mnemo:incident` (rigid, 7 phases)

Production fire workflow. Distinct from `mnemo:debug` because the
cadence is different (stabilize → investigate → RCA → file post-mortem)
and past-incident memory is gold.

| Phase | What |
|---|---|
| 1. Severity | User confirms sev. Stub `docs/incidents/<date>-<short-name>.md` with severity + start time. |
| 2. Pull priors | `mnemo query "<symptoms>"` filtered to `intent_tags=['incident', 'debug', 'feedback']`. Show every past similar incident with citations. |
| 3. Stabilize | Actions taken (rollbacks, kill switches, traffic shedding). Logged into the incident doc with timestamps. |
| 4. Investigate | Hypotheses + checks. Each hypothesis cites at least one mnemo node or rules out a related one. |
| 5. RCA | Root-cause statement. Required: what changed, why it broke, why we missed it. |
| 6. Post-mortem | Fill `docs/incidents/<date>-<name>.md` (timeline, RCA, action items). |
| 7. Memory | Promote durable lessons to `feedback_<topic>.md` entries via `mnemo:add-knowledge`. |

## 8. Distribution

| Component | Repo | Toolchain | Distribution |
|---|---|---|---|
| Daemon + Claude Code plugin | `mmct-jsc/mnemo` (existing) | Python (uv, pytest, ruff) | PyPI: `mnemo` package; plugin via Claude Code |
| Middleware | `mmct-jsc/mnemo` under `clients/middleware-py/` | Python (separate `pyproject.toml`) | PyPI: `mnemo-middleware` package |
| VS Code extension | `mmct-jsc/mnemo-vscode` (NEW) | TypeScript (esbuild, vsce) | GitHub release `.vsix` (no marketplace publish in v1.1) |

The middleware shares the daemon's repo because it shares the language
and benefits from the same lockfile pattern, CI image, and contributor
mental model. The VS Code extension gets its own repo because `vsce`
expects an extension-rooted directory and the TS toolchain doesn't
mix cleanly with the Python project.

Independent versioning: `mnemo` (daemon) and `mnemo-middleware` are
separate published packages on PyPI. They can advance versions
independently within the v1.1 cycle if needed.

### CI gates per phase

Each daemon phase (1-5) keeps the existing test suite green and adds
its own unit + integration tests in the same commit. Middleware (7-8)
and VS Code (9-10) ship their own CI. Release phase (11) regenerates
`uv.lock` and runs the full cross-platform matrix.

### Risk register

- VS Code marketplace publish needs a `VSCE_PAT` secret. v1.1 ships
  `.vsix` to GitHub releases only. Marketplace publish is a v1.2
  decision once the extension has been used in the wild.
- `pypdf` parse quality varies. Phase 4 ships with a `parse_quality`
  flag on the source row (`'high'|'low'`) so PDF nodes can be
  deprioritized in scoring without re-parsing.
- Provider SDKs change. Each shim is pinned to a tested SDK version
  range; CI runs nightly against latest to catch drift early.
- Active-project race when two IDEs are open. Mitigation: per-call
  `project_key` override is honored, so multi-IDE users can pin
  the project explicitly when they need to.

## 10. Hard rules (carry-over)

- No `Co-Authored-By` trailers on commits, ever.
- No emojis in code, docs, or commit messages unless the user
  explicitly asks.
- Conventional commit prefixes (`feat:`, `fix:`, `chore:`, `docs:`,
  `test:`, `refactor:`, `perf:`).
- Daemon binds to `127.0.0.1` only, never `0.0.0.0`.
- main is branch-protected. Push to `release/1.1.0`, open PR, merge in
  the web UI when CI is green.
- Use HEREDOC for multi-line commit messages.
