## 0. Why this release exists (the owner's challenge)

After v5.28.0 shipped, the owner challenged the whole retrieval paradigm:

> "how can we make this actually reliable if it's hit rate and query does
> not help at all, return data does not match or compatible with current
> AI agents approach?"

The honest answer (rethink doc S1-S2): **single-shot `hit@k` is the wrong
reliability bar.** It measures snippet retrieval -- the commodity axis where
an agent's own grep+read loop already wins. It never measured the two things
grep CANNOT do, which are mnemo's actual moat:

- **the typed graph** -- blast radius (reverse `calls`/`method_of`), route
  tracing, provenance (code -> commit -> memory);
- **cross-session memory** -- "we tried X in v4.6, it failed because Y."

You cannot improve what you cannot measure, and we have never measured the
moat. v6.0.0 builds that instrument FIRST (measure-before-tuning, the v5.26
lesson) and ships it alone.

## 2. North-star reframe (what "reliable" now means)

**Reliability = agentic task-success**: given an agent + mnemo's tools, can
the correct answer to a graph/memory question be reached in <= N tool calls?
Four question classes; three are mnemo's job to own:

| class | example question | mnemo organ measured |
|-------|------------------|----------------------|
| `structural` | "what calls `store.rekey_node`?" | the GRAPH (reverse `calls` edges) |
| `provenance` | "why does this function exist?" | the GRAPH (code -> commit -> memory walk) |
| `memory_recall` | "what did we decide about dual-remote ship?" | RETRIEVAL over memory nodes |
| ~~semantic code search~~ | "where is the BM25 search" | COMPLEMENT grep -- NOT owned, NOT measured here |

`structural` + `provenance` isolate the graph: the subject node is resolved
by its STABLE key (v5.28), so retrieval noise is removed and we measure the
graph's correctness + reachability directly. `memory_recall` is, by nature, a
retrieval task (the only path to a memory node is the query) -- this is where
`hit@k` lives on, but reframed as one task class among three, not the
headline.

## 3. What v6.0.0 actually measures: the deterministic REACHABILITY baseline

The harness mirrors the existing `eval_retrieval.py` seam exactly: **the
harness is dependency-free; the caller owns execution via a `solve_fn`
callback** (cf. `run_entries(entries, query_fn=...)`). v6.0.0 ships ONE
solver -- a deterministic ORACLE solver that walks the canonical tool path
for each class against the live store:

- `structural`: resolve subject by stable key -> `get_edges(dst_id, "calls")`
  -> caller keys. (graph correctness; 1 graph op)
- `provenance`: resolve subject by stable key -> bounded BFS over out-edges
  -> the commit / memory keys that explain it. (graph reachability)
- `memory_recall`: `retrieve.query(prompt)` -> top-k node keys. (retrieval; 1 op)

**Why an oracle, not an LLM, first.** The oracle answers "is the moat data
CORRECT and REACHABLE through the actual tools, and at what tool-call cost?"
-- the honest FLOOR under any agent. If the oracle can't recover a known
answer, no LLM agent can: that is a mnemo bug (a missing `calls` edge, a
broken provenance path), and it is exactly the class of defect `hit@k` never
surfaced. The oracle is CI-safe (no API key, deterministic) and PINS the moat
against regressions -- nothing currently does. The real LLM-in-the-loop
agent-success number is the natural follow-up and drops into the SAME
`solve_fn` seam (S7).

**Non-circularity (critical).** The ground-truth answer for each task is
established INDEPENDENTLY of the tool being scored: for `structural` /
`provenance` it is hand-verified by reading the source / commit / memory
during curation (S/step 4); the oracle then walks the graph's edges. If the
parser missed a call edge, oracle recall < 1.0 -- a REAL measured gap, not a
tautology. (Generation may PROPOSE the answer from the graph, but a human
pins it; that is what "hybrid" buys us.)

## 4. New module API -- `daemon/mnemo/eval_tasks.py`

```python
TASK_CLASSES = ("structural", "provenance", "memory_recall")

@dataclass
class Task:
    id: str                 # stable slug, e.g. "struct-rekey-callers"
    cls: str                # one of TASK_CLASSES
    prompt: str             # NL question (for the deferred LLM-agent solver)
    subject_key: str | None # stable key of the subject node (graph classes)
    answer_keys: list[str]  # GROUND TRUTH: stable keys / memory names
    budget: int = 2         # max meaningful tool calls allowed
    note: str = ""

@dataclass
class TaskResult:
    task: Task
    found_keys: list[str]
    calls_used: int
    recall: float           # |answer & found| / |answer|
    precision: float        # |answer & found| / |found|
    f1: float
    success: bool           # answer_keys subset-of found_keys AND calls<=budget

# pure helpers (no store dependency -> trivially unit-testable)
def load_task_set(path) -> list[Task]
def score_task(found_keys, task) -> TaskResult        # set match + budget
def run_tasks(tasks, *, solve_fn) -> list[TaskResult] # caller owns solve_fn
def aggregate_tasks(results) -> dict   # per-class {n, success_rate, avg_calls, avg_recall, avg_precision} + overall
def format_task_report(results, agg, corpus=None) -> str  # LEADS with per-class moat success

# store-backed (the v6.0.0 oracle solver + the generator)
def resolve_key(store, key) -> Node | None            # suffix match on normalized source_path
def generate_tasks(store, *, per_class=10) -> list[Task]   # hybrid candidate generator
def oracle_solve(store, task, *, embedder=None, k=5) -> tuple[list[str], int]
```

Reuse `eval_retrieval._norm` (backslash->slash, strip line-range, lowercase)
for all key comparisons so scoring is consistent with the retrieval harness.
Success policy: `set(norm(answer_keys)) <= set(norm(found_keys))` AND
`calls_used <= budget`. Precision/F1 reported but not gating (a complete
answer that also pulls neighbours is still a success; precision flags noise).

## 6. Surfaces

- **CLI**: new `mnemo eval-tasks` command -- mirrors `eval_cmd`: load fixture,
  build a dispatching `solve_fn` (daemon-first `retrieve.query` for
  `memory_recall`; in-process store walk for graph classes), print
  `format_task_report` + a pinned `corpus_snapshot`.
- **No MCP / HTTP / UI surface change.** This is a developer/measurement
  instrument, like `mnemo eval`. The 30-tool MCP contract stays byte-stable
  (no snapshot regen). This keeps the blast radius minimal -- exactly the
  "harness first" intent.

## 7. Explicitly deferred (the "then decide" follow-up)

- **LLM-agent solver** -- a real agent loop over the MCP tools, opt-in via env
  flag (mirror the analyzer-judge / T9-judge pattern). Drops into the
  `solve_fn` seam; produces the true agent-success number above the oracle
  floor.
- **Injection-as-pointer** (cut A) -- lean hook output (titles + ids + <=2
  highest-confidence lessons). Decision already locked (S1); built next.
- **Navigable-refs audit** (cut C) -- verify `mnemo_query`/`mnemo_traverse`
  expose `file:line` + ids cleanly; optional next-hop affordance.
- **Ranking lever** -- code-node type-boost + snake_case tokenization (the
  secondary `hit@k` axis), only if it still matters after the reframe.

```

[provenance]    n=6   success=1.00  avg_calls=1.0  recall=1.00  precision=0.32

overall         n=26  success=0.92  avg_calls=1.0
```

**This baseline empirically validates the reframe.** The GRAPH moat is the
strong, reliable part; RETRIEVAL is the weak link:

- **structural 1.00 / precision 1.00** -- reverse-`calls` blast radius is
  perfectly reachable AND clean. The thing grep cannot cheaply do, mnemo nails.
- **provenance 1.00 recall / 0.32 precision** -- the introducing commit is
  always recovered, but the walk returns ~3x extra co-linked nodes
  (README/CLAUDE docs, sibling commits). FINDING: provenance is RECALL-strong
  but PRECISION-noisy -> a future lever (rank the walk by edge confidence /
  relevance).
- **memory_recall 0.75** -- 2/8 real misses, both honest retrieval failures:
  - `mem-release-workflow`: the dedicated `feedback_mnemo_release_workflow`
    node lost top-5 to `reference_mnemo_pipelines` (term overlap).
  - `mem-handover-v528`: a CODE node (`cli.py::migrate_code_identity`)
    outranked the v5.28 handover memory -> the prose/code ranking issue =
    exactly the deferred secondary lever (S4). (precision is naturally low
    here: single-answer tasks over k=5.)

### Triage of the structural/provenance 1.00

These two classes pin answers DERIVED from the graph (oracle-verified
candidates), so 1.00 confirms the oracle path is internally consistent + the
graph is reachable -- it is a REGRESSION GUARD, not independent proof. The
strongest INDEPENDENT signal is memory_recall (answers pinned to known memory
filenames, decided before any retrieval), where the honest number is 0.75.

### Discovered moat GAP (recorded, not faked): mnemo-own provenance is empty

Probing the live graph for provenance candidates whose subject is under
`/daemon/mnemo/` returned ZERO: no mnemo-own code node has a
`references_function` edge to a commit carrying a `motivated_by`/`closed_by`
memory. So "why does `rekey_node` exist?" is NOT answerable from the graph
today, even though the real answer (the v5.28 commits + handover) exists. The
6 provenance fixture tasks therefore use other indexed repos (stock_trade /
TS app) whose commits name their README/CLAUDE. ROOT CAUSE candidates for a
follow-up: (a) the knowledge-base repo's commits may lack the doc-name /
trailer signals `motivated_by`/`closed_by` require; (b) self-repo commit
ingestion coverage. This is the harness's first real finding -> v6.x backlog.
