## 0. Context

The user stopped the feature march: mnemo ships 20+ green releases yet is
invisible -- "no commands show up, no notify." Research (last session +
this one) found the root cause and verified the fix contract against the
live Claude Code (CC) plugin system. This release makes mnemo **actually
install + register** in Claude Code. Discoverability/presence (workstream
B) and trust/precision (workstream C, folds in the perf arc) follow.

**This release's job:** a fresh user (or an AI on their behalf) runs ONE
documented path and afterward the `/mnemo-*` commands appear in Claude
Code and the SessionStart hook fires. `mnemo doctor` proves every link
green.

## 1. The verified CC plugin/marketplace contract (evidence-backed)

Sources: official `plugin-dev` plugin installed on this machine
(`~/.claude/plugins/marketplaces/claude-code-plugins/plugins/plugin-dev/
skills/plugin-structure/references/manifest-reference.md` +
`skills/mcp-integration/SKILL.md`), real installed examples
(`superpowers-dev/.../hooks/hooks.json`, `pixel-plugin/.../.mcp.json`),
`~/.claude/settings.json` `enabledPlugins`, `installed_plugins.json`, and
the `claude-code-guide` agent cross-checking the official docs.

- **marketplace.json** at `.claude-plugin/marketplace.json` (sibling to
  `plugin.json`). Required: `name`, `owner{name}`, `plugins[]`. Each
  plugin entry requires `name` + `source`; may carry the plugin.json
  metadata fields.
- **Single-repo plugin is supported**: plugin entry `"source": "./"` means
  "the plugin is at the marketplace root." Proven on disk (superpowers-dev).
- **Install flow**: `/plugin marketplace add mmct-jsc/mnemo` then
  `/plugin install mnemo@mnemo`. Writes `enabledPlugins["mnemo@mnemo"]=true`
  in `~/.claude/settings.json` + an entry in
  `~/.claude/plugins/installed_plugins.json`. Name resolution is
  `<plugin>@<marketplace>`.
- **Local/dev install** (verify before publishing): `/plugin marketplace
  add <local repo path>` then `/plugin install mnemo@mnemo`, or
  `claude --plugin-dir <path>`. Commands appear immediately;
  `/reload-plugins` after editing non-skill components.
- **Component discovery**: `./commands/`, `./agents/`, `./skills/`,
  `./hooks/hooks.json`, `./.mcp.json` are **auto-discovered** from default
  locations. Manifest path fields only SUPPLEMENT defaults and **must
  start with `./`** (forward slashes, no `../`, no absolute).
- **Hooks schema** (authoritative): `plugin.json` `hooks` is a string path
  (default `./hooks/hooks.json`) OR an inline object. The object is NESTED:
  ```json
  { "EventName": [ { "matcher": "...", "hooks": [
      { "type": "command", "command": "<shell command>", "timeout": 30 } ] } ] }
  ```
  There is **NO `platforms` key**. The `command` is a full shell command
  string; `${CLAUDE_PLUGIN_ROOT}` expands to the plugin dir.
- **MCP bundling exists** (`mcpServers` in plugin.json OR `./.mcp.json`),
  auto-starts on enable, BUT plugin-provided MCP tools are renamed
  `mcp__plugin_<plugin>_<server>__<tool>`. See section 4.

## 2. What is broken today (all confirmed this session)

mnemo sits unregistered at `~/.claude/plugins/mnemo` (a plain dir) -- absent
from `enabledPlugins` AND `installed_plugins.json` (both list 12 OTHER
plugins). On top of that root cause, the packaging itself is malformed:

- **No `marketplace.json`** -> cannot be installed the supported way.
- **`plugin.json` is malformed 3 ways**: `"commands": "commands/"` and
  `"skills": "skills/"` violate the "must start with `./`" rule (and both
  dirs are auto-discovered, so the keys should be omitted); the `hooks`
  block uses a **fictional `platforms` key** and the **wrong flat shape**
  (`{command, platforms}`) instead of nested `{matcher, hooks:[{type,
  command}]}`. So even if registered, the hooks would not fire.
- **Hooks are doubly fragile**: `user-prompt-submit.sh` / `post-tool-use.sh`
  shell out to **`python3`**, which on Windows is usually `python`; and the
  `.sh`/`.ps1` split has no valid cross-platform selector in the manifest.
- **Installers are obsolete**: `install.sh`/`.ps1` symlink the repo into
  `~/.claude/plugins/` and say "restart CC" -- but CC ignores unregistered
  directories. Hooks fail-OPEN, so a half-install is silent.
- **The test suite encodes the bug**: `test_plugin_manifest.py` asserts the
  invented `platforms` key + 6 `.sh`/`.ps1` scripts; `test_install.py`
  asserts the symlink. Green CI validated mnemo's OWN schema, not CC's
  (lesson #134 made concrete).
- **Retrieval precision (workstream C evidence, reproduced)**: dogfood query
  "how does the mnemo CLI implement the status command + check daemon
  health" (answer = `daemon/mnemo/cli.py:292-311`) returned 8
  session-handover memory nodes, scores flat 0.50-0.56, ZERO code nodes,
  not `cli.py`. "How/where code" questions never surface the answering
  file. (Not fixed here; recorded for workstream C.)

## 3. The design -- six deliverables

### 3a. `.claude-plugin/marketplace.json` (NEW)
Single-repo marketplace, plugin source `"./"`:
```json
{
  "name": "mnemo",
  "owner": { "name": "mnemo contributors",
             "url": "https://github.com/mmct-jsc/mnemo" },
  "metadata": { "description": "<one-line>", "version": "5.24.0" },
  "plugins": [ {
    "name": "mnemo", "source": "./",
    "description": "<plugin.json description>",
    "version": "5.24.0", "license": "MIT",
    "homepage": "https://github.com/mmct-jsc/mnemo",
    "keywords": ["memory", "graph-rag", "claude-code", "knowledge", "mcp"]
  } ]
}
```
No personal email (use repo `url`).

### 3b. `.claude-plugin/plugin.json` (FIX -> documented-fields-only)
Reduce to the documented, valid fields:
- Keep `name` ("mnemo", matches kebab regex), `version`, `description`,
  `author`, `license`, `homepage`, `keywords`.
- **Drop** `"commands"` and `"skills"` (auto-discovered; bare strings are
  invalid). **Drop** the non-standard `config` block (unused by the hooks;
  CC may reject unknown keys -- minimize first-install risk). [Verify
  `config` is unreferenced before dropping; grep `daemon/` + `hooks/`.]
- Replace the inline `hooks` block with the file reference
  `"hooks": "./hooks/hooks.json"` (see 3c).

### 3c. `hooks/hooks.json` (NEW) + `mnemo hook <event>` CLI (NEW); delete the 6 scripts
The cross-platform fix: one shell command per event that routes through
the `mnemo` binary (already on PATH after install), so no `platforms` key,
no `python3` dependency, no `.sh`/`.ps1` duplication, and the hook logic
becomes tested Python.

`hooks/hooks.json`:
```json
{ "hooks": {
  "SessionStart":     [ { "matcher": "startup|resume|clear",
      "hooks": [ { "type": "command", "command": "mnemo hook session-start" } ] } ],
  "UserPromptSubmit": [ { "matcher": "",
      "hooks": [ { "type": "command", "command": "mnemo hook user-prompt-submit" } ] } ],
  "PostToolUse":      [ { "matcher": "Edit|Write",
      "hooks": [ { "type": "command", "command": "mnemo hook post-tool-use" } ] } ]
} }
```
New CLI group `mnemo hook` (in `cli.py`, a `typer.Typer` like `source`/`node`):
- `mnemo hook session-start` -> emit the memory-map. Output the documented
  SessionStart contract: JSON `{"hookSpecificOutput": {"hookEventName":
  "SessionStart", "additionalContext": "<map>"}}` (verify exact shape in
  impl; raw stdout is the fallback). Fails open (exit 0, no output) if the
  store/daemon is unavailable.
- `mnemo hook user-prompt-submit` -> read stdin JSON `{prompt}`, run
  `retrieve.query(..., budget=800, k=5, exclude_local_only=False)`, emit
  citations as `additionalContext`. Fails open.
- `mnemo hook post-tool-use` -> read stdin JSON `{tool_input:{file_path}}`,
  and if the path matches `*/memory/*.md | */CLAUDE.md | */docs/plans/*.md`,
  trigger a detached `reindex --no-embed`. Emits nothing.
Port the existing `.sh` logic verbatim into these subcommands, then
**delete** `hooks/*.sh` + `hooks/*.ps1`.

### 3d. `install.sh` / `install.ps1` (REWRITE)
Keep steps 1-3 (python/uv check, `uv sync`, PATH shim) + step 5 (`mnemo
init`). **Replace** step 4 (symlink into `plugins/`) with:
- (a) Best-effort MCP registration: if `claude` is on PATH, run
  `claude mcp add mnemo -- mnemo mcp` (idempotent; skip if already listed).
  Else print the command.
- (b) Print the two CC-internal commands the user runs inside Claude Code
  (plugin enablement is a CC action, not a file copy):
  `/plugin marketplace add mmct-jsc/mnemo` then `/plugin install mnemo@mnemo`.
- (c) Detect a stale `~/.claude/plugins/mnemo` from the old installer and
  WARN to remove it (do not auto-delete).
Final message: "run `mnemo doctor` to verify everything is wired."

### 3e. `mnemo doctor` (NEW CLI command) -- the loud end-to-end verifier
Replaces silent fail-open. Factor checks into pure functions
`check_*(env) -> CheckResult(name, ok: bool|None, detail, hint)` (testable
against synthetic dirs); the command renders a ✓/✗/? list and exits 1 if
any check is ✗. Checks:
1. `mnemo` resolvable on PATH (`shutil.which`).
2. daemon reachable (`GET http://127.0.0.1:7373/v1/health` -> version);
   hint `mnemo daemon start`.
3. index present (`store.count_nodes()` sum > 0); hint `mnemo reindex`.
4. MCP registered: best-effort `claude mcp list` parse for "mnemo"; `?` +
   manual-verify note if `claude` not on PATH; hint
   `claude mcp add mnemo -- mnemo mcp`.
5. plugin registered: `~/.claude/settings.json` `enabledPlugins` has a
   `mnemo@*` == true AND `installed_plugins.json` has a `mnemo@*` entry;
   hint the two `/plugin` commands.
6. plugin manifest self-valid: bundled `marketplace.json` + `plugin.json`
   parse, required fields present, all component paths start with `./`,
   `hooks.json` parses with the nested shape (regression guard).

### 3f. `skills/mnemo-setup/SKILL.md` (NEW) -- AI-executable install
A skill the agent activates on "install mnemo" / "set up mnemo here" /
"make mnemo work". Body steps: detect OS/host -> run `install.sh`/`.ps1`
(engine: deps + PATH + init) -> if Claude Code, `claude mcp add mnemo --
mnemo mcp` -> tell the user to run `/plugin marketplace add mmct-jsc/mnemo`
+ `/plugin install mnemo@mnemo` (the agent cannot type slash commands) ->
run `mnemo doctor` -> report the ✓/✗. Satisfies "have an AI install it."

## 4. The MCP decision -- keep it registered SEPARATELY (not plugin-bundled)

**Decision: do NOT bundle the MCP server in the plugin for v5.24.0.** MCP
stays registered via `claude mcp add mnemo -- mnemo mcp` (run by the
installer + the setup skill, verified by `mnemo doctor`).

Why not bundle (the rejected `./.mcp.json` option):
- **Tool-name churn**: plugin-provided MCP tools are renamed
  `mcp__plugin_mnemo_mnemo__mnemo_query` (vs the working
  `mcp__mnemo__mnemo_query`). Bundling would break every host guide, the
  29-tool wire snapshot, and the user's working tools -- for no user-
  visible gain (the tools already resolve).
- **The venv can't live in the plugin cache**: marketplace install COPIES
  the repo to `~/.claude/plugins/cache/...` with no `.venv` and no deps
  (pixel-plugin bundles a prebuilt `.exe`; mnemo's engine is a
  hundreds-of-MB Python venv). So a plugin MCP `command` would have to call
  a PATH `mnemo` anyway -- i.e. it is NOT self-contained even if bundled.
Given both, separate registration is strictly simpler and lower-risk. The
user's actual pain (commands/notify) is the PLUGIN half, fixed by 3a-3c
regardless. Revisit bundling only if a future "one-command everything"
goal outweighs the churn.

This makes the clean split explicit: **installer = engine** (venv, PATH,
`claude mcp add`, init); **plugin (marketplace) = CC wiring** (commands,
hooks, skills); **`mnemo doctor` = verify both halves are present + linked.**
