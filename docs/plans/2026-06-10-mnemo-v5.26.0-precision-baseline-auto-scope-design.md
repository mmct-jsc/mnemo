# mnemo v5.26.0 -- precision baseline + auto-scoped retrieval (workstream C, part 1)

> DoD-first design doc (2026-06-10). First of three releases in workstream C
> (TRUST & PRECISION). Per pipeline #21; rides with release/5.26.0.
> C part 2 (v5.27.0) = exactness boost (BM25/RRF/chunking), measured against
> THIS release's baseline. C part 3 (v5.28.0) = stable code-node identity +
> reindex speed (lesson #129).

## 1. Goal

Two of the four C work-items, sequenced measure-first:

1. **Precision@k harness** -- a labelled query set + scorer, so every later
   retrieval change is judged by a number, not vibes. No tuning without a
   baseline.
2. **Fix the cross-project leak via AUTO-SCOPING** (user spec, 2026-06-10):
   "auto detect when working in a project in the current IDE to query
   properly; if not indexed yet, also ask the user if they want to index
   and use properly."

## 2. Root cause of the leak (investigated, confirmed)

`retrieve.query` ALREADY isolates: `project_isolation_mode` defaults to
"strict" (hard-drops cross-project nodes when `active_project` is set;
`project_key=None` cross-cutting nodes survive per the v1.2.1 fix; BASE
bypasses). **The leak is that no production caller passes `active_project`**:
the UserPromptSubmit hook, the MCP `mnemo_query` default, and ad-hoc CLI
queries all send None -> unscoped -> the CGE/Argos node outranked mnemo's own
files in a mnemo question. The fix is plumbing + UX, not scoring.

## 3. Components

### Component 1 -- retrieval eval harness (`mnemo eval` + labelled set)

- **Labelled set**: `daemon/tests/fixtures/retrieval_eval.json` -- ~15-25
  entries: `{prompt, expect_source_contains: [substr...], project_key?,
  note}`. Two flavors:
  - SEEDED entries (deterministic): used by the unit-mode test against a
    synthetic corpus built in-test (FakeEmbedder) -- assert scoping +
    exact-name behavior structurally.
  - SELF entries (dogfood): real "where/how is X in mnemo" questions whose
    answering files are known (e.g. "where is the statusline formatted" ->
    `statusline.py`; "how do hooks inject memory" -> `cli.py`;
    "where are MCP tools registered" -> `agent_tools.py`).
- **Scorer** (`daemon/mnemo/eval_retrieval.py`, pure): given (hits, expect)
  compute hit@k / precision@k / MRR; aggregate over the set.
- **CLI `mnemo eval`**: runs the SELF set against the LIVE store + real
  embedder (daemon-first like the hook), prints per-query hit@5 + aggregate
  P@1/P@5/MRR, exits 0 always (a report, not a gate). This is the baseline
  instrument for v5.27.0.
- **Unit tests**: scorer math; seeded-corpus run with FakeEmbedder proves the
  harness plumbing; the eval fixture file parses + entries well-formed.
  (CI never downloads the model: live mode is local-only by usage, not flag.)

### Component 2 -- auto-scoped queries (the leak fix)

Single helper, used by every entry point:

```python
def resolve_auto_scope(store, cwd: str | None) -> tuple[str | None, bool]:
    """(project_key_to_use, is_indexed). Auto-derive the active project from
    cwd via paths.resolve_project_key. Apply it ONLY when that project
    actually has nodes (else scoping a fresh project would strict-filter
    everything to zero); when it has none -> (None, False) so callers can
    surface the index-me offer."""
```

- **UserPromptSubmit hook**: CC's payload carries `cwd`. Resolve ->
  `_daemon_query(prompt, project_key=...)` (QueryIn already accepts it) and
  the in-process fallback passes `active_project=...`.
- **MCP server (`mnemo_query`)**: when the caller does not pass
  `project_key`, default to the auto-scope derived from the server process
  cwd (hosts spawn stdio servers in the project dir). Explicit `project_key`
  always wins. Same has-nodes guard.
- **CLI `mnemo query`**: `--project` still explicit; add `--auto-scope`
  default ON deriving from cwd (with `--no-auto-scope` escape).
- Response/output gains `scope: {project_key, auto: bool}` so the scoping is
  visible (trust through transparency).

### Component 3 -- "not indexed yet -> offer to index" (user spec)

When `resolve_auto_scope` returns `is_indexed=False` for the cwd:

- **Session-start banner** (human-visible `systemMessage`): append
  ` | this project is not indexed -- run: mnemo source add <cwd>`.
- **additionalContext** (model-visible): add one line telling Claude it MAY
  offer to index the current project (`mnemo source add <cwd>` then
  `mnemo reindex`) if the user wants project-scoped recall -- the
  "ask the user" half happens through the agent, which is the only
  conversational channel a hook has.
- **mnemo_query first-call notice** (MCP-only hosts): when unindexed, the
  notice says so + names the command.
- NEVER auto-index: indexing a repo is a user decision (cost + privacy).

## 4. Anti-goals

- No scoring-weight changes in this release (that is v5.27.0, measured
  against this baseline). Retrieval math byte-stable for scoped=None calls.
- No auto-indexing without an explicit user action.
- Auto-scope must NEVER zero out results for an unindexed cwd (the has-nodes
  guard) and must never override an explicit project_key.
- Harness adds no CI model download; live eval is a local instrument.

## 5. DoD

1. `mnemo eval` prints a baseline P@1/P@5/MRR over the SELF set against the
   live corpus (numbers recorded in the handover for v5.27.0 to beat).
2. Live re-run of the original leak repro ("how does a user install mnemo
   ...") from this repo's cwd: the CGE/Argos node NO LONGER appears in the
   top hits (strict scoping active via auto-scope).
3. From an unindexed directory, `mnemo hook session-start` banner carries the
   index-me offer; from this repo it does not.
4. Suite green + ruff clean; MCP wire snapshot regenerated only if tool
   descriptions/params changed (mnemo_query gains no new REQUIRED param;
   project_key stays optional).
