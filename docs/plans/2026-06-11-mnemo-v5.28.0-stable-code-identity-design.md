# mnemo v5.28.0 -- stable code-node identity + reindex mtime-skip (workstream C, part 3; FINAL arc item)

> DoD-first design doc (2026-06-11). Closes lesson #129 and the
> make-it-real arc. Rides with release/5.28.0. Measured against the
> v5.27.0 `mnemo eval` baseline (expanded to >= 40 queries -- see SS5).

## 0. RISK BANNER -- read before any code

This is an **in-place migration over the user's real ~18k-node memory
DB**. A bug can corrupt accumulated memory. Non-negotiable guardrails:

- **No big-bang DELETE.** The migration re-keys legacy nodes in place
  (UPDATE source_path, preserve id + embedding). It never mass-deletes
  and re-creates.
- **Dry-run on a COPY first.** `mnemo migrate-code-identity --dry-run`
  runs the full reconcile against a *copy* of `mnemo.db` and reports
  re-keyed / would-orphan / ambiguous / id-preservation / edge-delta.
  The live reindex is only run after the dry-run on a copy is clean.
- **Reversible.** A source_path UPDATE that preserves the node id is
  reversible; we keep a `.v5.27-pre-migration` DB copy until verified.
- Gate on the expanded `mnemo eval` (>= 40): retrieval must not
  regress.

## 1. Problem (grounded; mnemo-cited)

Code declaration nodes are keyed by
`source_path = <file>:<start_line>-<end_line>`
([`CodeUnit`](../../daemon/mnemo/parsers/code.py) docstring, code.py:90-124).
Reconcile matches an existing node by *exact* `source_path`
(`get_node_by_source`, store.py:1049), called per parsed unit at
ingest.py:673. When code above a declaration shifts, its line range --
hence its key -- changes:

- `get_node_by_source(<new range>)` misses -> a fresh `Node.new()`
  (new `uuid4`, store.py:265) is created and **re-embedded**
  (ingest.py:678-693).
- The old-range node is not in `seen_source_paths` -> the deletion
  sweep removes it (ingest.py:794-804).

Net: +273 / -255 nodes per reindex (lesson #129). What churns:

1. **Re-embed cost** -- every shifted declaration is re-encoded by
   MiniLM even when its body is byte-identical. Dominant reindex cost.
2. **Node-id stability** -- the new node gets a new `uuid`, so anything
   keyed by node id and *not* regenerated from source is lost:
   `created_at` history, feedback events, `audit_queue` fingerprints,
   `query.retrieved_ids` backrefs, any hand-curated edge.

Derived code edges self-heal: edge intent (`defines` / `method_of` /
`imports` / `call_sites`) is JSON-encoded into each node's
`frontmatter_json` under a `code_unit` key (`parse_code_file`,
ingest.py:235-301) and rebuilt every reindex from the touched set
(`_resolve_code_edges` + `resolve_calls`, scope.py:87-134, `add_edge`
ON CONFLICT UPDATE). So edges are not the loss -- the re-embed and the
id-keyed history are.

### 1.1 Not all code nodes use the line-range key

The migration's "legacy key" detector must respect these existing forms:

- **module** -> bare file path (already stable; no change).
- **endpoint** -> `endpoint:METHOD:path` (already stable; `_endpoint_unit`,
  react.py:205-225).
- **route** -> `<file>:<start>-<end>#<method>` (django/express/flask/
  fastapi extractors).
- **function / class / method** -> `<file>:<start>-<end>` (the churn case).

### 1.2 Consumers of the `:start-end` suffix (must keep working)

Found via `mnemo_query` (one of these my grep missed):

1. git-log overlap, regex `:(\d+)-(\d+)(?:#.*)?$` (ingest.py:1041).
2. UI strip in base.html:256.
3. [`_line_range_of`](../../daemon/mnemo/extractors/fastapi.py) (fastapi.py:154-166,
   `rpartition(":")`) -- used at *extraction* time on a freshly-built
   declaration unit.

## 2. New identity scheme

Declaration key becomes `<file>::<qualified_name>` (double-colon
separator -- distinct from the legacy single-colon line suffix, so the
two forms are trivially discriminable):

| node type        | qualified_name              |
|------------------|-----------------------------|
| code_function    | `name`                      |
| code_class       | `name`                      |
| code_method (py/js) | `<ClassName>.<method>`   |
| code_method (go) | `<ReceiverType>.<method>`   |
| route            | `<handler_qualified_name>#<METHOD>` |

- **Collisions** (same qualified name in one file -- overloads,
  conditionally-defined, repeated `<anonymous>` arrow fns): append an
  occurrence ordinal `#<n>` for the 2nd+ occurrence in document order
  (first stays unsuffixed). This preserves the disambiguation the line
  range gave (`CodeUnit` docstring) without line sensitivity.
  - DECISION (accepted tradeoff): the ordinal is document-order based,
    so if two same-named decls swap order their keys swap. This is far
    rarer than a line shift, and on swap the bodies differ -> hash
    differs -> they update anyway. Documented, not solved.
- **Module / endpoint** keys unchanged (already stable).

### 2.1 Where the line range goes

`source_path` no longer carries the range. The range moves into the
existing nullable `frontmatter_json` under the `code_unit` block:
`{"line_start": int, "line_end": int, ...}`. **No schema change.**

The three suffix-consumers (SS1.2) read the range from frontmatter,
falling back to the legacy suffix parse when frontmatter lacks it (so
they keep working on not-yet-migrated rows mid-rollout):

- git-log: read `code_unit.line_start/end` per code node; fall back to
  the regex when absent.
- `_line_range_of`: the parser hands the extractor the line range
  directly (the extractor already has the AST node); the source_path
  parse becomes a fallback only.
- base.html: strip both `::<qual>` and the legacy `:start-end` for the
  display label.

## 3. Migration -- lazy reconcile-with-fallback (the safe path)

No separate destructive pass. Re-keying happens *in place* during the
normal reconcile, the first time each file is reindexed after upgrade:

1. Parser emits the stable key `<file>::<qualified_name>` + line range
   in frontmatter.
2. Reconcile (ingest.py:673) tries `get_node_by_source(stable_key)`.
3. **Miss -> fallback match before creating new:** look up the legacy
   node for *this same declaration*:
   - candidates = nodes whose `source_path` matches the legacy
     line-range form for this file (`<file>:` prefix, optional
     `#method`) AND `type == parsed.type` AND `name == parsed.name`.
   - exactly one candidate -> **re-key in place**: UPDATE its
     `source_path` to `stable_key`, write the line range into
     frontmatter, keep the id; re-embed only if `hash` changed.
   - multiple candidates (overload case) -> pick the one whose stored
     legacy range is nearest the new unit's `line_start` (deterministic).
   - zero candidates -> genuine new node (today's path).
4. Legacy nodes never matched (declaration truly deleted) fall through
   to the existing orphan sweep unchanged.

New store method: `rekey_node(node_id, new_source_path, line_start,
line_end)` -- single-row UPDATE of `source_path` + frontmatter, id and
vec row untouched. Pure, unit-testable, reversible.

### 3.1 Dry-run harness (the gate)

`mnemo migrate-code-identity --dry-run [--db <copy>]`:

- opens a COPY of the store read-mostly, runs the reconcile match logic
  WITHOUT committing,
- reports: `rekeyed`, `would_orphan`, `ambiguous`, `new`, plus an
  id-preservation assertion (every legacy code node maps to exactly one
  surviving id) and an edge-count delta (should be ~0 net, since edges
  regenerate).
- DoD requires: on a copy of the *live* DB, `would_orphan` contains
  only declarations actually removed from source, `ambiguous` is small
  and hand-explainable, and id-preservation holds for the rest.

## 4. Reindex mtime-skip (speed)

Skip parsing a file whose mtime hasn't advanced since its last index --
but **only after** its code nodes are on stable keys (else a skipped
file never migrates). Ordering:

- Track per-file mtime (new `file_index` table: `path PK, mtime,
  indexed_at`), written when a file is parsed.
- Skip a file iff: mtime unchanged AND no legacy-keyed code node
  remains for it (migration done). First post-upgrade reindex of a file
  always parses (to migrate); subsequent unchanged reindexes skip.
- Skipped files still add their existing nodes' source_paths to
  `seen_source_paths` so the deletion sweep doesn't reap them.

DECISION: mtime-skip is scoped to `code_repo` sources only this release
(memory/docs files are cheap and benefit from frontmatter re-parse).

## 5. `mnemo eval` expansion (measurement gate)

The n=14 SELF set is too noisy to tune against (v5.27.0 finding: corpus
drifts under measurement). v5.28.0 expands it:

- >= 40 labelled queries spanning memory + code + how/where phrasings
  (deliberately include the "how does X work" shape that currently
  returns pure prose -- SS1, dogfood).
- Pin a corpus snapshot (record node count + a content hash in the eval
  report header) so two runs are comparable.
- `mnemo eval` stays a report, not a CI gate (unchanged contract).

## 6. Anti-goals

- No new scoring weights; no embedding/model change.
- No schema-breaking migration (frontmatter_json reused; the one new
  table `file_index` is additive via `_ensure_columns`-style create).
- Not solving the prose-outranks-code ranking miss in THIS release --
  the eval expansion *measures* it; a code-node type-boost for how/where
  queries is a follow-up (the dogfood in SS1 reconfirmed the need).

## 7. DoD

1. Dry-run on a copy of the live DB: id-preservation holds; `would_orphan`
   is only genuinely-deleted code; numbers recorded here + in the handover.
2. Live reindex after migration: code-node count is **stable** across two
   consecutive no-op reindexes (the churn metric -- target |delta| ~ 0 vs
   today's +273/-255).
3. Re-embed count on a no-op reindex drops to ~0 for unchanged files.
4. Expanded `mnemo eval` (>= 40, pinned snapshot) does **not regress**
   hit@5 vs the v5.27.0 baseline.
5. All three suffix-consumers (SS1.2) verified against migrated rows.
6. Full suite green; ruff clean. A `.v5.27-pre-migration` DB copy kept
   until DoD #2-#4 confirmed live.

## 8. Build order (TDD; one commit per step)

1. Parser: stable-key construction + line range into frontmatter
   `code_unit` (code.py + the 4 framework extractors). Unit: same decl
   keeps key across a line shift.
2. Store: `rekey_node` (pure single-row UPDATE, id-preserving) + a
   `find_legacy_code_node(file, type, name)` lookup. Unit: re-key
   preserves id, vec row, edges.
3. Reconcile: fallback-match-and-rekey in ingest before new-node create.
   Unit: a moved function is re-keyed, not churned (id stable, no
   re-embed).
4. Suffix-consumers: git-log + `_line_range_of` + base.html read range
   from frontmatter with legacy fallback. Unit each.
5. Dry-run CLI `mnemo migrate-code-identity --dry-run`. Unit: reports
   shape on a seeded legacy DB.
6. mtime-skip: `file_index` table + skip gate (code_repo only, post-
   migration only). Unit: unchanged file skipped, changed file parsed,
   legacy file not skipped.
7. `mnemo eval` expansion to >= 40 + pinned snapshot header.
8. LIVE: dry-run on a copy -> record -> live reindex -> verify DoD
   #2-#4 -> bump 5.27.0 -> 5.28.0 + CHANGELOG + ship.

## 9. Open decisions surfaced for review

- SS2 collision ordinal is document-order (accepted tradeoff) -- ok?
- SS4 mtime-skip scoped to code_repo only this release -- ok, or also
  memory/docs?
- Migration is lazy (re-key on first touch) vs an explicit one-shot
  `mnemo migrate-code-identity --apply`. Recommendation: lazy is the
  primary path (safer, incremental); the `--apply` form is just the
  dry-run harness with commit enabled, offered as an opt-in for users
  who want to migrate in one controlled run. Both share one code path.
